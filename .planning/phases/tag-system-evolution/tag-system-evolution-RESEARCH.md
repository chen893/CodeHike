# Phase 1: Tag System Evolution - Research

**Researched:** 2026-04-19
**Domain:** Tag system architecture, controlled vocabulary, multi-dimension filtering, tag relations, user follows
**Confidence:** HIGH

## Summary

This phase merges three roadmap versions (VNext, V+2, V+3) into a single unified execution. The tag system currently has a complete skeleton (DB tables, AI generation, API routes, Explore UI) but suffers from 10 identified bugs and lacks structural dimension (tagType), controlled vocabulary enforcement, tag relations, and user engagement features.

The codebase follows a clean Service/Repository pattern where tag data access lives in `tag-repository.ts`, business orchestration in `tag-service.ts`, and AI generation in `tag-generator.ts`. All existing code is reusable -- the phase extends rather than rewrites. The critical dependency chain is: schema migration (add tagType) -> vocabulary seeding -> AI constraint enforcement -> Explore tab UI -> tag relations -> user follows.

Key technical risks: (1) Chinese slug generation is completely broken (`\w` regex strips all CJK characters), requiring `pinyin-pro` library integration; (2) search + tag filter mutual exclusion in `explore-service.ts` must be resolved with a unified query builder; (3) the merged phase is large (26-36 days estimated separately), so execution ordering and wave boundaries are critical.

**Primary recommendation:** Execute in 4 waves strictly following the dependency chain: Wave 1 = bugs + schema + tagType backfill + vocabulary seeding; Wave 2 = AI constraint + Explore tab UI + candidate queue; Wave 3 = tag detail page + tag relations + management API; Wave 4 = user follows + "My Follows" page.

**What:** All DB access goes through repositories; business logic in services; pages/handlers call services only.
**When to use:** Every new data access or mutation in this phase.
**Example:**

### Anti-Patterns to Avoid

- **Direct DB access from pages/route handlers:** AGENTS.md explicitly forbids calling `lib/repositories/*` or `lib/db/*` from `app/*`. Always go through services.
- **Blocking analytics calls:** Never `await` analytics in render paths. Keep fire-and-forget pattern.
- **Client-side data fetching outside feature clients:** All client fetch calls must go through feature client files (e.g., `tags-client.ts`, `follows-client.ts`), not scattered in view components.
- **IN() subquery for tag filtering:** Current `tutorial-search-repository.ts` uses IN(array of IDs) which doesn't scale. Replace with JOIN-based filtering.
- **Free-form AI tag generation:** Must be replaced with vocabulary-constrained selection. LLMs will "hallucinate" tags not in the vocabulary -- always validate output.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Chinese slug generation | Custom regex/unicode handling | `pinyin-pro` library | Edge cases: polyphonic characters, rare CJK, mixed CJK+ASCII. pinyin-pro handles all correctly. |
| SQL query building for multi-tag filtering | String concatenation of WHERE clauses | Drizzle ORM query builder with `and()` / `inArray()` | SQL injection risk, complex nesting, and Drizzle already imported |
| AI output validation | Custom JSON parsing | Zod schema + Vercel AI SDK `Output.object()` | Already established pattern; handles edge cases, partial output, type safety |
| Cache invalidation | Manual cache key management | Next.js `unstable_cache` with `revalidateTag` | Framework-provided; works with serverless; already recommended in codebase analysis |
| Tag co-occurrence calculation | Custom matrix library | SQL GROUP BY + COUNT with self-join | PostgreSQL handles this efficiently; no external dependency needed |

**Key insight:** The existing codebase already has mature patterns for AI output (Output.object + Zod), data access (Drizzle ORM), and client-server communication (feature clients). Every new feature should follow these patterns rather than introducing new approaches.

## Common Pitfalls

### Pitfall 1: Chinese Slug Collision After Fix
**What goes wrong:** After fixing `generateSlug` to support Chinese via pinyin, multiple Chinese tags may produce the same pinyin slug (e.g., "前端" and "钱端" both become "qian-duan").
**Why it happens:** Pinyin is a lossy conversion; multiple Chinese characters share the same pronunciation.
**How to avoid:** In `getOrCreateTag`, when slug collision occurs, append a numeric suffix (e.g., `qian-duan-2`). The existing unique constraint + 23505 error handling already catches this; just need to update the slug generation fallback.
**Warning signs:** `getOrCreateTag` 23505 errors spike after slug fix deployment.

### Pitfall 2: LLM Vocabulary "Hallucination"
**What goes wrong:** AI returns tags that are close to but not exactly matching vocabulary entries (e.g., "React.js" when vocabulary has "React").
**Why it happens:** LLMs do not strictly adhere to enum constraints in free-form output, even with structured output mode.
**How to avoid:** Add a post-processing validation step that fuzzy-matches AI output to vocabulary (Levenshtein distance or simple includes/startsWith). Reject unmatched tags to candidate queue. Never trust AI output blindly.
**Warning signs:** More than 20% of AI-generated tags go to candidate queue.

### Pitfall 3: Migration Lock-in with pgEnum
**What goes wrong:** Adding a new value to pgEnum (e.g., adding a 4th tagType) requires a separate ALTER TYPE statement, which can't be combined with other schema changes in the same transaction in some PostgreSQL versions.
**Why it happens:** PostgreSQL enum types are separate database objects; ALTER TYPE is DDL that may require table locks.
**How to avoid:** The three tagType values (technology, category, level) are locked by D-01. If future changes are needed, create a new enum and migrate. For this phase, a single migration with CREATE TYPE + ALTER TABLE ADD COLUMN is safe.
**Warning signs:** Migration takes > 5 seconds on production DB.

### Pitfall 4: Search + Tag Filter Regression
**What goes wrong:** When unifying search and tag filter paths, the combined query may perform worse than either path alone.
**Why it happens:** JOIN-based tag filtering combined with full-text search `tsvector` may produce complex query plans that PostgreSQL optimizer handles poorly.
**How to avoid:** Test with EXPLAIN ANALYZE on realistic data. Consider materializing tag counts separately. Keep the search path as a simple WHERE clause and add tag filtering via JOIN only when tag params are present.
**Warning signs:** Explore page load time exceeds 2 seconds after refactor.

### Pitfall 5: Candidate Queue Growing Without Bound
**What goes wrong:** AI generates many candidate tags that never get reviewed; candidate queue becomes a dumping ground.
**Why it happens:** Non-blocking design means no pressure to review candidates.
**How to avoid:** Add a scheduled cleanup (or manual trigger in management API) that deletes candidates older than 90 days. Include candidate count in tag health metrics.
**Warning signs:** Candidate table has more rows than vocabulary table.

### Pitfall 6: Backward Compatibility Break with URL Params
**What goes wrong:** Old URLs like `/explore?tag=react` break or show empty results after switching to `?technology=react`.
**Why it happens:** The URL parameter scheme changes from generic `tag` to type-specific `technology`/`category`/`level`.
**How to avoid:** In the server component, check for `params.tag` first. If present, look up the tag's tagType and redirect (301 or server-side map) to the corresponding typed parameter. This is explicitly required by D-13.
**Warning signs:** Analytics show 404s or empty results for `/explore?tag=*` URLs after deployment.

### Pitfall 7: tagType Backfill AI Accuracy
**What goes wrong:** AI misclassifies tags (e.g., putting "React Hooks" into category instead of technology).
**Why it happens:** The boundary between technology and category can be ambiguous for some tags.
**How to avoid:** Target >= 70% coverage (D-05), not 100%. Tags the AI can't confidently classify stay NULL. Include the tagType definitions and examples in the backfill prompt. Manual review of a sample (10-20 tags) to estimate accuracy before marking the job complete.
**Warning signs:** Post-backfill, category has 100+ tags (should be 8-15 per D-02).

## Code Examples

### Chinese Slug Fix with pinyin-pro
```typescript
// tag-repository.ts - REPLACE generateSlug function
import { pinyin } from 'pinyin-pro';

function generateSlug(name: string): string {
  const lower = name.toLowerCase().trim();

  // Convert Chinese characters to pinyin
  const pinyinResult = pinyin(lower, {
    toneType: 'none',    // No tone marks
    type: 'array',       // Return as array
    separator: '-',      // Join with hyphens
  });

  const slug = pinyinResult
    .join('-')
    .replace(/[^a-z0-9\u4e00-\u9fff-]/g, '')  // Keep alphanumeric + CJK + hyphens
    .replace(/-+/g, '-')
    .replace(/^-+|-+$/g, '')
    .slice(0, 64);

  return slug || 'tag';
}
```

```typescript
export async function getExploreData(options: {
  page?: number;
  sort?: string;
  technology?: string;   // NEW
  category?: string;     // NEW
  level?: string;        // NEW
  tag?: string;          // BACKWARD COMPAT
  lang?: string;
  search?: string;

  const tagsPromise = tagRepo.listAllTags();

  // Build tag filters from typed params

  if (options.category) tagFilters.push({ tagSlug: options.category, tagType: 'category' });
  if (options.level) tagFilters.push({ tagSlug: options.level, tagType: 'level' });

  // Backward compat: old ?tag=xxx maps to first match

    total = result.total;

    const result = await searchRepo.listPublishedForExplore({
      page: options.page,
      pageSize: 20,
      sort: options.sort === 'popular' ? 'popular' : 'newest',
      tagFilters,
      lang: options.lang,
    });
    tutorials = result.tutorials;
    total = result.total;
  }
  // ...
```

```typescript
  fromTagId: uuid('from_tag_id')
    .notNull()
    .references(() => tutorialTags.id, { onDelete: 'cascade' }),
  toTagId: uuid('to_tag_id')
    .notNull()
    .references(() => tutorialTags.id, { onDelete: 'cascade' }),
  relationType: varchar('relation_type', { length: 32 }).default('co_occurrence'),
  strength: sql<number>`coalesce(strength, 0)`,  // co-occurrence count or weight
  computedAt: timestamp('computed_at', { withTimezone: true }).defaultNow(),
}, (table) => [
  uniqueIndex('tag_relations_unique_pair').on(table.fromTagId, table.toTagId),
]);

// User follows
export const userTagFollows = pgTable('user_tag_follows', {
  userId: text('user_id')
    .notNull()
    .references(() => users.id, { onDelete: 'cascade' }),
  tagId: uuid('tag_id')
    .notNull()
    .references(() => tutorialTags.id, { onDelete: 'cascade' }),
  createdAt: timestamp('created_at', { withTimezone: true }).defaultNow(),
}, (table) => [
  primaryKey({ columns: [table.userId, table.tagId] }),
]);
```

### Co-occurrence Calculation SQL
```sql
-- Calculate tag co-occurrence: tags that appear together on the same tutorial
INSERT INTO tag_relations (from_tag_id, to_tag_id, relation_type, strength)
SELECT
  ttr1.tag_id AS from_tag_id,
  ttr2.tag_id AS to_tag_id,
  'co_occurrence' AS relation_type,
  COUNT(*) AS strength
FROM tutorial_tag_relations ttr1
JOIN tutorial_tag_relations ttr2
  ON ttr1.tutorial_id = ttr2.tutorial_id
  AND ttr1.tag_id < ttr2.tag_id  -- avoid duplicate pairs
GROUP BY ttr1.tag_id, ttr2.tag_id
ON CONFLICT (from_tag_id, to_tag_id)
DO UPDATE SET strength = EXCLUDED.strength, computed_at = NOW();
```

## Open Questions

1. **Tag management API auth model**
   - What we know: Current auth only checks `session?.user?.id` existence. No role system exists.
   - What's unclear: Should management API (merge/delete/rename) be admin-only, or should any tutorial owner be able to manage tags on their tutorials?
   - Recommendation: For this phase, implement admin-only management API by adding a `role` field to users table (minimal implementation). Ordinary tag editing via PUT remains login-required.

2. **"My Follows" page route**
   - What we know: CONTEXT.md says "independent page" (D-16). Route location is Claude's discretion.
   - What's unclear: Best route location that doesn't conflict with existing `/[slug]` dynamic route.
   - Recommendation: `/following` is clean and unambiguous. Must verify it doesn't match any existing tutorial slug (RESERVED_SLUGS check).

3. **Candidate queue implementation**
   - What we know: D-08 says "new table or status='candidate'". Both approaches work.
   - What's unclear: Which approach is better for query performance and code simplicity.
   - Recommendation: Use a separate `tag_candidates` table rather than adding a status column to `tutorial_tags`. Reasons: (1) keeps the main tags table clean for vocabulary queries, (2) candidate tags are conceptually different from approved tags, (3) avoids NULL status ambiguity with existing NULL tagType tags.
