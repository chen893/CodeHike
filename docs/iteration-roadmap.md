# VibeDocs 迭代路线图

**创建日期：** 2026-04-13
**最后更新：** 2026-04-13（v3.5 后路线图规划）
**基线版本：** v3.5（用户系统 + 公开首页 + 分享 + 部署自动化 + 版本快照）
**当前分支：** docs/iteration-roadmap
**整合来源：** 现状解析、架构演进、用户价值、增长闭环 4 个 Agent 的分析结论

---

## 当前系统所处阶段

VibeDocs v3.5 已完成从 0 到 1 的全链路建设并达到可对外部署状态。

**已交付能力（v3.3-v3.5 全量）：**

- 完整 DSL（含多文件）+ Assembler 渲染链路 + Scrollytelling 渲染器
- 可视化 Diff + 实时 Patch 校验 + 代码选区快捷操作
- 多模型 Provider 注册表 + Prompt-Provider 解耦 + 模型探测
- 生成风格模板 + 首次体验模板 + 增量重新生成 + 错误恢复增强
- 用户系统（NextAuth v5 + GitHub OAuth）+ 数据隔离 + 路由保护
- 公开首页（SEO + OG meta + JSON-LD）+ 教程分享 + embed + OG 图片
- 取消生成按钮 + 草稿版本快照
- CI/CD 部署自动化 + 监控工具 + 测试覆盖
- 20+ 个 API 端点

**当前最大问题（4 个 Agent 共识）：**

- 获取能力极弱（增长闭环 Agent: 3/10），教程不可被搜索引擎有效发现，没有病毒传播机制
- 可测量性几乎为零（增长闭环 Agent: 1/10），没有埋点，不知道谁在看、从哪来
- AI 生成可靠性约 50 分（用户价值 Agent），首次失败率仍然偏高，大源码不支持
- 产品验证度低（现状解析 Agent: ~15%），没有真实用户使用数据

**产品所处阶段：** Late MVP / Pre-Seed -- 工程完成度 ~85%，产品验证度 ~15%。系统完整但缺乏用户获取和验证手段。

## 路线图主线

```
v3.3  编辑可用性 + 工程基础          ← 让工作区从"能用"变成"好用"        [已完成]
v3.4  AI 生成升级 + 首次体验优化      ← 突破 token 天花板，降低首次失败率  [已完成]
v3.5  产品化 & 对外可用              ← 从个人工具变成可部署可分享的产品    [已完成]
v3.6  增长基础设施 + 生成可靠性       ← 让产品可被发现、可测量、更可靠
v3.7  教程发现 + 标签体系            ← 建立教程浏览和发现能力
v3.8  激活与复访                     ← 优化新用户转化 + 建立回访动力
v4.0  高级创作 + 平台基础             ← 导出、系列、协作等高级功能
```

**排序逻辑：**

1. v3.6 先做增长基础设施（最小埋点 + 阅读 CTA + 导出），因为当前连"用户做了什么"都不知道，无法做任何数据驱动的决策。同时提升 AI 生成可靠性，这是产品价值的根基。
2. v3.7 做教程发现和标签体系，有了埋点数据后才能衡量发现页面的效果。这是获取用户的核心手段。
3. v3.8 做激活与复访，有了发现页面带来流量后，需要优化新用户转化率和回访率。
4. v4.0 在用户基础稳固后做高级创作功能，这需要更强的数据模型和更多基础设施。

## v3.3 -- 编辑可用性 + 工程基础 [已完成]

> 让编辑工作区从"技术验证"变成"日常可用"。

### 核心目标

Patch 编辑是用户与教程 DSL 交互的核心操作。当前 patch 编辑是纯文本 find/replace，无实时校验、无可视化 diff、修改后不知道对不对。v3.3 的目标是让编辑工作区具备基本的日常可用性。

### 关键交付

- 可视化 Diff + 实时 Patch 校验（CodeDiffView + usePatchValidation）
- 代码选区快捷操作（Selection API + 浮动菜单）
- Patch 中间态预览
- 取消发布功能
- 测试覆盖（patch-chain、assembler、layer-boundary）
- 错误分类框架

---

## v3.4 -- AI 生成升级 + 首次体验优化 [已完成]

> 突破 DeepSeek 8192 token 硬天花板，提升生成质量和首次体验。

### 核心目标

DeepSeek 8192 token 上限是当前 AI 生成的硬天花板，限制了教程长度和源码规模。v3.4 引入多模型支持解除天花板，通过生成风格模板和首次体验模板提升 AI 输出质量。

### 关键交付

- Provider 注册表（`lib/ai/provider-registry.ts`）
- Prompt-Provider 解耦层（`lib/ai/prompt-adapters.ts`）
- 模型配置 schema + 模型选择 UI + 模型能力探测
- 生成风格模板（对话式 / 教科书式 / 渐进式）
- 首次体验模板（预设 Redux 示例源码）
- 增量重新生成 + 从失败步骤批量重试

## v3.5 -- 产品化 & 对外可用 [已完成]

> 从个人工具变成可部署、可分享、有用户的产品。

### 核心目标

v3.5 集中做三件事：用户系统（数据隔离）、公开首页（获客入口）、分享能力（传播渠道），让产品达到可对外展示和使用的状态。

### 关键交付

- 用户认证（NextAuth v5 + GitHub OAuth + JWT sessions + Edge-compatible middleware）
- DB schema 扩展（userId + NextAuth tables）
- API 认证中间件 + 路由保护
- 教程展示首页（OG meta + JSON-LD + SEO）
- 教程分享对话框 + embed 端点 + OG 图片生成
- 取消生成按钮
- 草稿版本快照（draft_snapshots 表）
- CI/CD 部署自动化（GitHub Actions）
- 监控工具（`lib/monitoring/metrics.ts`）

## v3.6 -- 增长基础设施 + 生成可靠性

> 让产品可测量、可传播、更可靠。从"能部署"到"能增长"的关键跳跃。

### 核心目标

v3.5 完成后产品已可对外使用，但存在两个致命缺口：**不可测量**（没有埋点，不知道谁在看、从哪来）和**传播断裂**（读者看完教程后没有"我也要创建"的引导）。同时 AI 生成可靠性仍是用户价值的短板（用户价值 Agent 评估约 50 分）。v3.6 集中解决这三个问题。

### 解决的核心矛盾

**"产品已上线但无法知道是否有效"** -- 没有数据就无法做任何后续决策。埋点是所有增长策略的前置条件。阅读端 CTA 是最轻量的病毒传播机制（增长闭环 Agent: K 从 0.05 提升到 0.2 的关键手段）。

### 前置依赖

- v3.5 用户系统（埋点需要识别用户身份）
- v3.5 公开首页和分享能力（埋点需要覆盖这些新页面）
- v3.5 监控工具（`lib/monitoring/metrics.ts` 可复用和扩展）

### 关键能力

#### P0：最小埋点体系

| 编号 | 功能 | 说明 |
|------|------|------|
| 3.6.1 | 埋点基础设施 | `lib/monitoring/analytics.ts` -- 轻量事件记录，服务端调用，写 PostgreSQL `events` 表 |
| 3.6.2 | 核心事件定义 | 3 个起步事件：(1) `tutorial_viewed` 教程被阅读 (2) `tutorial_created` 教程被创建 (3) `cta_clicked` CTA 被点击 |
| 3.6.3 | DB events 表 | `id`, `event`, `userId?`, `payload jsonb`, `createdAt` -- 最简 schema，不加索引先 |
| 3.6.4 | 阅读页埋点 | `app/[slug]/page.jsx` 服务端记录 `tutorial_viewed`，payload 含 slug、referrer |
| 3.6.5 | 创建转化埋点 | 创建成功后记录 `tutorial_created`，含来源（首次体验模板 / 空白创建） |

#### P0：阅读端 CTA

| 编号 | 功能 | 说明 |
|------|------|------|
| 3.6.6 | 教程尾部 CTA 组件 | `components/tutorial/create-cta.tsx` -- 教程滚动到底部后展示"用你的源码创建教程"卡片 |
| 3.6.7 | 创建页 referrer 透传 | CTA 链接带 `?ref=cta&source={slug}`，创建页接收并传入埋点 |

#### P1：Sitemap + 导出

| 编号 | 功能 | 说明 |
|------|------|------|
| 3.6.8 | Sitemap 生成 | `app/sitemap.ts` -- `generateSitemap()` 从 DB 查询所有已发布教程 slug，生成标准 XML sitemap |
| 3.6.9 | `robots.txt` | `app/robots.ts` -- 允许所有爬虫，指向 sitemap URL |
| 3.6.10 | 导出 Markdown | `lib/services/export-markdown.ts` + `app/api/tutorials/[slug]/export/markdown/route.ts` -- 将 TutorialDraft 转为可下载 Markdown（代码块 + 文案段落） |
| 3.6.11 | 导出 HTML | `lib/services/export-html.ts` + `app/api/tutorials/[slug]/export/html/route.ts` -- 生成独立 HTML（内嵌 CSS + 高亮样式，无外部依赖） |

#### P1：AI 生成可靠性提升

| 编号 | 功能 | 说明 |
|------|------|------|
| 3.6.12 | Patch 自动修复 | `lib/ai/patch-auto-fix.ts` -- 当 patch 的 find 匹配失败时，用模糊匹配（trim + 空白归一化）尝试自动修复 |
| 3.6.13 | 生成后自动校验修复 | 修改 multi-phase-generator：step-fill 完成后如果 patch 失败，先尝试自动修复再重试 |
| 3.6.14 | 大源码预处理 | `lib/ai/source-preprocessor.ts` -- 当源码超过阈值时自动截断非核心文件，优先保留主文件 + 入口文件 |

### 不做什么

- 教程发现/搜索页面（v3.7）
- 标签系统（v3.7）
- 教程系列/多章节（v4.0）
- 实时协作（远期）
- 用户通知系统（v3.8）
- 深度 SEO 优化（后续版本按需迭代）
- 第三方分析工具接入（先用自建，够用再说）

### 技术架构要求

- 埋点走服务端，不依赖客户端 JS 执行。在 Server Component 和 Route Handler 中直接调用
- events 表先不加索引，数据量上来后再按 `event` + `createdAt` 加索引
- 导出功能复用 assembler 的 payload 构建，不另起一套渲染
- Patch 自动修复在 `lib/tutorial/draft-code.js` 中实现，作为 `applyContentPatches` 的容错层
- Sitemap 使用 Next.js 内置 `generateSitemaps` API

### 主要风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 埋点数据量增长导致 events 表膨胀 | 查询变慢 | 初期用最简 schema，预估日均 1000 条以下；加 TTL 清理策略（90 天） |
| Patch 自动修复可能引入错误匹配 | 教程内容错乱 | 仅做 trim + 空白归一化，不做语义匹配；修复后仍然需要唯一匹配 |
| 导出 Markdown 格式选择 | 用户期望不一致 | 先做标准 CommonMark，支持代码块 + 标题层级；不做 PDF（依赖重） |
| 大源码预处理可能丢失关键代码 | 生成质量下降 | 截断策略保守：优先保留主文件完整 + 其他文件只保留函数签名和注释 |

### 成功标准

- 3 个核心埋点事件上线，能在管理界面或 SQL 查询中看到数据
- 阅读端 CTA 上线，创建页能接收 ref 参数
- Sitemap 可被 Google Search Console 提交和索引
- 至少 1 个导出格式（Markdown）可用
- AI 生成成功率从当前水平提升 10%+（通过 `generationQuality` 数据衡量）
- 单版本工作量控制在 1-2 周

## v3.7 -- 教程发现 + 标签体系

> 建立教程浏览和发现能力，让用户能找到感兴趣的教程。

### 核心目标

v3.6 的埋点和 SEO 基础设施就位后，v3.7 建立教程发现能力。当前已发布教程只能通过直接链接访问，没有浏览、搜索、分类的入口。教程发现是获客的核心手段（增长闭环 Agent: 获取从 3/10 提升的关键路径）。

### 解决的核心矛盾

**"教程发布了但没人能找到"** -- 只有直接链接传播，没有主动发现渠道。搜索引擎需要 sitemap（v3.6 已提供）和内容丰富的发现页面才能有效索引。

### 前置依赖

- v3.6 埋点体系（衡量发现页面的流量和转化效果）
- v3.6 Sitemap + robots.txt（搜索引擎索引基础）
- v3.5 公开首页（已有教程列表展示，需要增强为发现页面）

### 关键能力

#### P0：教程浏览页

| 编号 | 功能 | 说明 |
|------|------|------|
| 3.7.1 | 公开浏览页 `/explore` | 展示所有已发布教程，支持分页。卡片布局：标题、描述、语言标签、步骤数、阅读时间 |
| 3.7.2 | 首页整合发现入口 | 首页 `/` 增加搜索框或"浏览全部教程"入口，引导用户进入 `/explore` |
| 3.7.3 | 教程排序 | 按发布时间（默认）、按阅读量（依赖 v3.6 埋点数据） |

#### P0：标签系统

| 编号 | 功能 | 说明 |
|------|------|------|
| 3.7.4 | DB schema 扩展 | `tutorial_tags` 表：`id`, `name` (unique), `slug` (unique), `createdAt`；`tutorial_tag_relations` 表：`tutorialId`, `tagId` |
| 3.7.5 | 标签自动生成 | 发布时 AI 分析教程内容生成 3-5 个标签。复用已有 provider-registry + 轻量 prompt |
| 3.7.6 | 标签手动编辑 | 创作者可在编辑工作区修改标签 |
| 3.7.7 | 按标签筛选 | `/explore?tag=javascript` 筛选特定标签的教程 |

#### P1：搜索能力

| 编号 | 功能 | 说明 |
|------|------|------|
| 3.7.8 | 标题/描述全文搜索 | PostgreSQL `ts_vector` 全文搜索，支持标题 + 描述 + 标签。不引入 Elasticsearch |
| 3.7.9 | 语言筛选 | `/explore?lang=typescript` 按编程语言筛选 |

#### P1：创作者个人页

| 编号 | 功能 | 说明 |
|------|------|------|
| 3.7.10 | 用户公开主页 `/@username` | 展示用户的已发布教程列表，个人简介。最低可用版本：头像 + 名称 + 教程卡片列表 |
| 3.7.11 | 用户资料编辑 | 名称、简介、头像的编辑功能 |

### 不做什么

- 教程评分/收藏（远期）
- 推荐算法（远期，先靠标签和搜索）
- 教程系列（v4.0）
- 协作编辑（远期）
- 深度个性化（远期）
- 复杂搜索（Elasticsearch、向量搜索等）

### 技术架构要求

- 全文搜索用 PostgreSQL 原生 `ts_vector` + GIN 索引，不引入新依赖
- 标签自动生成复用 `lib/ai/provider-registry.ts`，用轻量 prompt（不占主生成流程）
- 浏览页使用 Server Component + `cache()` 策略，标签和搜索结果设短缓存（60s）
- 标签表和数据模型变更需要新建 migration
- 用户公开主页需要 `users` 表增加 `username` 字段（unique, nullable 先兼容）

### 主要风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 教程数量太少（初期可能 < 20 篇） | 发现页面看起来空洞 | 首期可以用示例教程填充；发现页设计为"教程少也好看"的布局 |
| 标签自动生成质量不稳定 | 标签不一致 | 允许手动编辑；发布前预览标签，不强制接受 |
| 全文搜索中文分词问题 | 中文教程搜索效果差 | PostgreSQL `pg_trgm` 扩展支持模糊匹配；中文先用 `simple` 分词器，后续按需升级 `zhparser` |
| username 字段需要用户主动设置 | `@username` 路由可能无数据 | 设为 nullable，未设置时用 userId 作为 fallback |

### 成功标准

- `/explore` 页面可访问，支持按标签和语言筛选
- 标签系统上线，新发布的教程自动生成标签
- 搜索框可用，能按标题和描述找到教程
- v3.6 的埋点数据显示 `/explore` 带来的创建转化
- 单版本工作量控制在 1-2 周

## v3.8 -- 激活与复访

> 优化新用户转化漏斗，建立回访动力，让用户从"来一次"变成"持续用"。

### 核心目标

v3.7 的发现页面会带来流量，但增长闭环 Agent 指出激活漏斗尚未优化。v3.8 的核心目标是：(1) 优化新用户从"首次访问"到"创建第一个教程"的转化率 (2) 建立让已创建过教程的用户回来的动力。

### 解决的核心矛盾

**"用户来了但没有留下来"** -- 激活和留存是最容易被忽视但最影响增长的环节。在获客渠道（v3.7）建立后，必须同步建设转化和留存能力。

### 前置依赖

- v3.6 埋点体系（衡量漏斗各环节转化率）
- v3.7 发现页面（流量入口）
- v3.5 用户系统（识别新老用户）

### 关键能力

#### P0：新用户引导

| 编号 | 功能 | 说明 |
|------|------|------|
| 3.8.1 | 新用户引导流程 | 首次登录后展示简短引导（2-3 步）：(1) "粘贴你的源码" (2) "告诉 AI 你想教什么" (3) "预览你的教程" |
| 3.8.2 | 空状态引导 | `/drafts` 列表为空时展示引导卡片而非空白页面，含"用示例源码试一次"快捷入口 |
| 3.8.3 | 首次体验模板优化 | 增强 v3.4.F 的首次体验模板：增加 2-3 个预设模板（不同语言/场景），降低首次使用决策成本 |

#### P1：创作者仪表盘

| 编号 | 功能 | 说明 |
|------|------|------|
| 3.8.4 | 教程数据面板 | 创作者能看到自己教程的阅读量、CTA 点击数。利用 v3.6 埋点数据聚合展示 |
| 3.8.5 | 草稿续写提醒 | `/drafts` 列表页高亮"未完成的草稿"，鼓励用户继续编辑 |

#### P1：通知基础

| 编号 | 功能 | 说明 |
|------|------|------|
| 3.8.6 | 站内通知系统 | `notifications` 表 + 通知列表 UI。初期通知类型：教程被阅读 50/100/500 次、教程被引用 |
| 3.8.7 | 邮件通知（可选） | 集成邮件服务（Resend/SendGrid），周报："你的教程本周被 N 人阅读"。不阻塞版本交付 |

### 不做什么

- 移动端 App / PWA（远期）
- 付费/订阅系统（远期）
- 教程系列（v4.0）
- 实时协作（远期）
- 复杂的推荐/个性化算法（远期）
- Push 通知（移动端，远期）

### 技术架构要求

- 引导流程用客户端组件 + localStorage 标记已完成步骤
- 创作者仪表盘用 Server Component 聚合 v3.6 events 表数据
- 通知系统用新 `notifications` 表（`id`, `userId`, `type`, `payload jsonb`, `read boolean`, `createdAt`）
- 邮件通知如做，选轻量服务（Resend），不自建邮件服务

### 主要风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 引导流程打断用户自然流程 | 用户反感 | 引导必须是可跳过的、非阻塞的，用户可随时关闭 |
| 仪表盘数据量少 | 看起来空洞 | 数据量少时展示鼓励性文案而非零值；示例："你的第一个教程已有 5 次阅读！" |
| 邮件服务需要额外环境变量 | 增加部署复杂度 | 设为可选功能，没有邮件配置时跳过邮件通知 |

### 成功标准

- 新用户引导流程上线，埋点可追踪引导完成率
- 空状态页面有引导内容而非空白
- 创作者仪表盘能看到教程阅读数据
- 至少 1 个触发型通知上线（阅读里程碑）
- 单版本工作量控制在 1-2 周

## v4.0 -- 高级创作 + 平台基础

> 在用户基础稳固后，引入高级创作功能，从"单篇教程工具"向"教程平台"演进。

### 核心目标

v3.6-v3.8 完成后，产品具备了可发现、可测量、有增长闭环的基础。v4.0 引入高级创作能力（教程系列、多语言生成、导出增强）和平台基础（协作、模板市场），满足进阶用户需求。

### 解决的核心矛盾

**"单篇教程工具的天花板"** -- 用户需要将多篇教程组织成系列，需要多语言版本，需要团队协作。这些是"重度创作者"从"试用"转为"依赖"的关键功能。

### 前置依赖

- v3.6 埋点体系（衡量高级功能的使用率）
- v3.7 标签系统（系列教程的发现依赖标签）
- v3.8 激活与复访（确保有足够的活跃用户验证高级功能）

### 关键能力

#### P0：教程系列

| 编号 | 功能 | 说明 |
|------|------|------|
| 4.0.1 | 系列数据模型 | `tutorial_series` 表（id, userId, title, description, slug, createdAt）；`series_items` 表（seriesId, tutorialId, sortOrder） |
| 4.0.2 | 系列创建和管理 | 创作者可创建系列、添加/移除教程、调整顺序 |
| 4.0.3 | 系列阅读页 | `/series/[slug]` 展示系列目录 + 首篇教程。含"上/下一篇"导航 |

#### P0：多语言生成

| 编号 | 功能 | 说明 |
|------|------|------|
| 4.0.4 | 多语言教程变体 | 同一教程可生成不同语言版本（中/英/日等），共享源码但文案不同 |
| 4.0.5 | 语言切换 UI | 阅读端和发现页支持语言切换 |

#### P1：导出增强

| 编号 | 功能 | 说明 |
|------|------|------|
| 4.0.6 | PDF 导出 | 基于已有 HTML 导出，用 Puppeteer/Playwright 生成 PDF（Vercel 上需考虑 runtime 限制） |
| 4.0.7 | 导入 Markdown | 从 Markdown 文件反向生成教程草稿（逆向解析） |

#### P1：协作基础

| 编号 | 功能 | 说明 |
|------|------|------|
| 4.0.8 | 教程共享编辑 | 基于角色（owner/editor/viewer）的共享机制。初期用简单的邀请链接 |
| 4.0.9 | 评论系统 | 读者可在教程步骤上留下评论。初期用简单文本评论 |

#### P2：平台化

| 编号 | 功能 | 说明 |
|------|------|------|
| 4.0.10 | 教程评分 | 1-5 星评分 + 简短评价 |
| 4.0.11 | 教程收藏 | 用户可收藏教程，在个人主页展示收藏列表 |
| 4.0.12 | 模板市场 | 创作者可分享教程模板，其他用户可基于模板创建 |

### 不做什么

- 实时协作编辑（需要 WebSocket + OT/CRDT，架构成本极高）
- 交互式练习自动生成（AI 能力边界外的功能）
- 学习者自适应（需要大量用户行为数据，远期）
- 付费/订阅/创作者收益（商业模式未验证前不做）
- 移动端 App

### 技术架构要求

- 教程系列需要新的数据表和 repository，但不改变现有 TutorialDraft 结构
- 多语言变体在 `published_tutorials` 表增加 `language` 字段，同一 slug 允许多语言版本
- 协作基础用邀请链接 + 角色 enum，不引入实时同步
- 评论系统需要 `comments` 表，通知系统（v3.8）扩展新通知类型
- PDF 导出考虑用 `@vercel/og` 或外部 API（浏览器 Puppeteer 在 serverless 上限制多）

### 主要风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 教程系列数据模型可能需要 TutorialDraft 结构变更 | 回归风险 | 只加外键关联，不改 TutorialDraft 本身 |
| 多语言生成增加 AI 成本 | 成本翻倍 | 按需生成，不自动翻译全部教程 |
| PDF 导出在 Vercel serverless 上受限 | 功能不可用 | 先用 HTML 导出 + 浏览器打印；或引入外部 PDF 服务 |
| 协作功能引入权限管理复杂度 | 安全风险 | 初期只做邀请链接模式，不做公开协作 |

### 成功标准

- 教程系列可创建、可阅读、可按序浏览
- 多语言变体可生成和切换
- 至少 1 个高级导出格式可用
- 协作邀请链接可工作
- 单版本工作量控制在 2-3 周（功能较多，可能拆为 v4.0 + v4.1）

## 近期不该做的事（v3.6-v3.8 期间）

以下功能在 v3.6-v3.8 期间明确不做，原因标注：

| 功能 | 不做的原因 |
|------|-----------|
| 实时协作编辑 | 需要 WebSocket + OT/CRDT 架构，成本极高，没有用户基础验证需求 |
| 大源码 AST chunking | 鸡生蛋问题：需要大源码用户验证，但没有大源码用户。先用大 context 模型缓解 |
| 付费/订阅系统 | 商业模式未验证，没有用户基础验证付费意愿 |
| 移动端 App / PWA | 核心创作场景在桌面端，移动端阅读体验已通过响应式布局覆盖 |
| 教程市场/平台化 | 需要 100+ 篇教程和活跃用户群才有效 |
| 学习者自适应 | 需要大量用户行为数据训练，远期功能 |
| 交互式练习生成 | AI 能力边界外，可靠性无法保证 |
| 深度 SEO（Schema.org 富摘要等） | 先用基础 SEO（sitemap + meta），等有流量数据后再迭代 |
| 第三方分析工具（Google Analytics、Mixpanel） | 先用自建埋点，数据量和复杂度不需要第三方工具 |
| GraphQL API | 当前 REST API 端点数量有限，没有过度获取问题 |
| 国际化 UI（多语言界面） | 创作端先支持中英，不做 UI 层面的 i18n |

## 附录 B：实施中的关键约束（提醒）

- **渲染层不可重写** -- `TutorialScrollytelling` 组件和 assembler 是固定基础设施
- **AI SDK v6 参数差异** -- `maxOutputTokens`（不是 `maxTokens`）
- **必须用 `@ai-sdk/openai-compatible`** -- 不用 `@ai-sdk/openai`（走 `/responses` 端点不兼容 DeepSeek）
- **混合 JS/TS** -- 渲染链路是 JS，新增功能是 TS，不强行统一
- **客户端不做 patch 计算** -- 所有 patch 应用、高亮、focus/marks 注入在服务端
- **新建代码遵循分层约定** -- 页面/API 调用 services，services 调用 repositories，不跨层
- **客户端请求收口 feature client** -- 不在视图组件里直接 fetch
- **生成耗时 30-60 秒** -- 需要合理的进度反馈和超时处理
- **NextAuth middleware 用轻量级实例** -- 无 DB adapter，Edge Runtime 兼容
- **埋点走服务端** -- 在 Server Component 和 Route Handler 中直接调用，不依赖客户端 JS
- **PostgreSQL 全文搜索** -- 不引入 Elasticsearch，用 `ts_vector` + `pg_trgm`
