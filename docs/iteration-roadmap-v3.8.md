# VibeDocs 后续版本路线图

> 基于 v3.8 完成状态的分阶段产品路线图。由 6 个独立分析 Agent（现状解析、架构约束、用户价值、增长闭环、版本策略、风险依赖）独立输出后汇总而成。

---

## 0. 当前系统所处阶段

**VibeDocs v3.8 是一个功能完备的 AI 驱动 scrollytelling 源码教学教程生成器。**

核心创作闭环（创建 → AI 生成 → 编辑 → 预览 → 发布）已跑通。发现页、标签系统、用户档案、分享、导出、事件追踪等"平台雏形"功能已上线。但产品处于**"寻找 Product-Market Fit"阶段**——功能虽然丰富，但没有真实用户数据证明核心价值假设。

**当前最关键的未知数不是"接下来做什么"，而是"现在的东西有没有人真的想用"。**

### 当前系统能力概览

| 模块 | 状态 | 关键能力 |
|------|------|---------|
| 核心渲染层 | ✅ 成熟 | CodeHike scrollytelling、patch find/replace、focus/marks、多文件、StepRail、移动端交替布局 |
| AI 生成 | ✅ 可用 | 三阶段生成（outline → step-fill → validate）、SSE 进度、质量评估、多 provider、patch 自动修复、增量重生成 |
| 编辑工作区 | ✅ 完备 | 步骤/patch/focus/marks 全量编辑、代码预览、结构变更检测、步骤管理 |
| 草稿管理 | ✅ 完备 | 创建/列表/删除、同步状态追踪、版本快照 |
| 发布与分享 | ✅ 完备 | 发布/取消发布、冻结快照、分享对话框、embed、Markdown/HTML 导出 |
| 发现与搜索 | ✅ 基础 | 探索页、标签系统、全文搜索（simple 配置）、用户档案 |
| 认证 | ✅ 基础 | GitHub OAuth + JWT、路由保护、草稿所有权 |
| 监控 | ✅ 基础 | events 表、埋点 helpers、fire-and-forget |
| SEO | ✅ 基础 | OG 图片、JSON-LD、meta 元数据 |

### 当前主要短板

1. **AI 生成不够可靠**——patch 匹配失败是最常见的生成失败原因，DeepSeek 8192 token 上限制约长教程
2. **首次体验认知门槛高**——Teaching Brief 6 个必填字段对非专业用户是壁垒
3. **创作者反馈回路为零**——发布后不知道有没有人看、没有互动
4. **没有真实用户数据**——漏斗各环节无量化指标
5. **部署和测试基础薄弱**——migration 不完整、测试覆盖有限、无错误监控

## 1. 路线图主线

**从"工具可用"到"有人用"到"有人留"。**

```
v3.8（当前）
  已有：完整创作闭环 + 发现 + 标签 + 搜索 + 分享 + 埋点 + 导出
  缺的：生成不够可靠、不知道用户在哪、创作者没理由回来
  ↓
基础设施修复（v3.9 前置，1-3 天）
  部署可靠性 + 关键路径测试 + 埋点补全
  ↓
v3.9 — 核心体验加固（1-2 周）
  AI 生成质量冲刺 + 首次体验快速通道
  ↓
v3.10 — 获客验证与数据驱动（1-2 周）
  漏斗数据分析 + 阅读体验优化 + 种子内容
  ↓
v3.11 — 创作者留存与反馈回路（1-2 周）
  创作者仪表盘 + 基础互动 + 通知 + 续写引导
  ↓
v4.0 — （基于 v3.9-v3.11 数据再定）
  候选：教程系列、搜索增强、协作基础
```

**关键判断节点：** v3.11 完成后看两个数据：(1) 周活用户是否在增长 (2) 是否有 >= 5 个活跃创作者。两个都是 → 可以规划 v4.0 高级功能。不是 → 继续打磨，不做高级功能。

## 2. 基础设施修复（v3.9 前置）

> 不占独立版本，但必须在 v3.9 开始前完成。

### 2.1 部署可靠性

- **问题：** Drizzle migration 目录不完整（`0001_noisy_revanche` SQL 文件缺失），每次部署需手动补 SQL
- **做什么：**
  - 补齐缺失的 migration 文件或重新生成 schema baseline
  - 部署脚本加入 `drizzle-kit push` 或 migration 回放步骤
  - 加入部署前 schema 一致性校验
- **为什么现在做：** 后续版本涉及新表/新列，不修这个每次部署都可能翻车
- **工作量：** 1 天

### 2.2 关键路径测试

- **问题：** 13 个测试文件只覆盖纯函数和结构边界，不覆盖关键业务流程
- **做什么：**
  - 为"生成 → 编辑 → 发布"关键路径补集成级 smoke test
  - 为 API 端点补最基本的请求/响应契约测试
  - CI 中加入 `npm run build` 之外的测试阶段
- **工作量：** 1-2 天

### 2.3 埋点补全与基础指标看板

- **问题：** 事件类型已注册但很多从未被调用（`onboarding_started`、`template_used`）；漏斗关键环节无数据
- **做什么：**
  - 补全 `draft_created`、`tutorial_published`、`tutorial_shared` 事件
  - 让 `template_used`、`onboarding_started`/`onboarding_completed` 真正被调用
  - `/new` 页面读取 `ref` 参数并上报（归因 CTA 来源）
  - 写简单 admin 页面或 SQL 查询查看：新注册数、首次创建率、首次生成成功率、发布率
- **工作量：** 1 天

### 前置依赖

- 无，可立即开始

### 预期收益

- 后续版本可以安全部署和迭代
- 第一次获得"产品有没有人用"的量化数据

## 3. Phase 1：v3.9 — 核心体验加固

**核心目标：让核心价值从"勉强可用"变成"真正好用"，同时让新用户 60 秒内看到成品。**

**为什么现在做这个：**
- 核心功能（AI 生成教程）不可靠，任何获客和留存手段都是空转（架构 Agent + 策略 Agent 共识）
- 首次体验门槛高是漏斗最大断裂点（增长 Agent 数据：预计 60-80% 掉失率）
- 首次体验改进成本极低（1-3 天），可以和 AI 质量冲刺并行

### 3.1 AI 生成质量集中冲刺

**做什么：**

1. **基于 generationQuality 数据分析失败模式**
   - 哪类源码失败率高、哪一步 patch 出错多
   - 针对高频失败场景强化 prompt 约束
   - 引入更多测试源码（当前只有 Redux 一份测试数据）

2. **大源码分步生成策略**
   - 当源码超过阈值时，自动拆分为多轮生成
   - 解决 DeepSeek 8192 token 上限对长教程的硬约束

3. **source-preprocessor 正式接入生成管道**
   - v3.8 已建 `source-preprocessor.ts` 但未接入 prompt 流

4. **生成失败后的体验改善**
   - 区分 outline 失败 vs step 失败，提供精准的恢复入口
   - 生成失败时展示"常见原因"和"建议操作"而不是裸错误文本

**不做：**
- 不换 AI 模型（DeepSeek 当前是兼容性最好的选择）
- 不重构 AI 生成架构（三阶段链路已经合理）

**对技术架构的要求：**
- 不需要架构调整，只在 AI 层（prompt 模板 + generator）改动
- 新增测试源码不需要代码改动

**主要风险：**
- AI 生成质量提升是系统工程，不是单次 sprint 能完全解决的
- DeepSeek token 上限是硬约束，只能绕不能消除

**成功标准：**
- AI 生成成功率（三阶段全通过 + patch 全绿）>= 80%
- 大源码（> 300 行）生成成功率 >= 60%
- 管理后台展示按失败原因分类的质量报表

### 3.2 首次体验快速通道

**做什么：**

1. **首次用户自动预填模板**
   - 首次进入 `/new` 的用户（`localStorage` 判断）自动预填 `FIRST_EXPERIENCE_TEMPLATE`
   - 添加"一键生成"按钮，跳过手动填写直接进入生成
   - 生成完成后直接展示预览（而不是进入编辑器）

2. **CreateCTA 强化**
   - 文案中文化（当前是英文）
   - 添加 `ref` 参数归因链路（`/new?ref=cta&source={slug}`）
   - 考虑在教程阅读页更显眼的位置放置 CTA（不只是底部）

3. **生成等待期的预期管理**
   - 在 30-60 秒等待期间展示成品预览截图或动画

**不做：**
- 不做多步引导 overlay（太重，后续版本视数据再定）
- 不改编辑工作区布局

**对技术架构的要求：**
- 不需要架构调整
- 前端改动为主（`/new` 页面 + `CreateCTA` 组件 + 生成进度组件）

**主要风险：**
- 自动预填可能让回访用户困惑（需"清空重来"入口）

**成功标准：**
- 首次创建成功率（从进入 /new 到生成完成）比当前提升 >= 50%
- `template_used` 事件有真实数据
- CTA 点击归因链路打通

### 3.3 版本范围控制

| 包含 | 不包含 |
|------|--------|
| AI prompt 优化 + 大源码分步 | 新产品功能（不新增页面） |
| 首次体验快速通道 | UI 改版 |
| 生成失败体验改善 | 移动端优化 |
| CTA 强化 + 归因 | 评论/互动系统 |

**总工作量：** 1-2 周

## 4. Phase 2：v3.10 — 获客验证与数据驱动

**核心目标：用数据验证获客管道是否有效，不是猜。**

**为什么现在做这个：**
- v3.9 让核心体验可靠了，现在可以放心引流
- v3.6-v3.7 建了发现页和埋点，但数据还没人看
- 需要验证"读者 → 创作者"转化假设是否成立

### 4.1 获客数据分析与行动

**做什么：**

1. **分析已有埋点数据**
   - 有多少人从 /explore 来？有多少人点了 CTA？
   - 转化漏斗长什么样？各环节掉失率多少？
   - 基于数据做针对性优化（搜索体验、CTA 位置、首页引导）

2. **SEO 第二轮**
   - 基于 Google Search Console 数据优化（不是盲猜）
   - 教程页面 meta description 自动生成
   - 按需增加 FAQ schema、HowTo schema

3. **搜索行为追踪**
   - 新增 `search_performed` 事件
   - 记录搜索关键词和结果数
   - 识别"零结果搜索"，了解用户想要但找不到的内容

### 4.2 阅读体验优化

**做什么：**

1. **基于阅读埋点数据做决策**
   - 滚动深度、停留时间、跳出率分析
   - 针对"看完就走"优化：底部推荐相关教程、阅读进度条、分享引导

2. **教程阅读完的轻量反馈**
   - 最后一步展示"这篇教程对你有帮助吗？"
   - 不做评论系统（内容审核成本高、用户量不够），只做最轻量的 yes/no

### 4.3 种子内容补充

**做什么：**
- 如果教程数量 < 30 篇，补充高质量种子教程
- 覆盖热门编程话题（React Hooks、TypeScript、Rust ownership 等）
- 用种子教程验证 AI 生成质量改进（v3.9）的效果

### 4.4 版本范围控制

| 包含 | 不包含 |
|------|--------|
| 埋点数据分析 + 行动 | 付费投放 |
| SEO 基于数据的优化 | 社交媒体自动化 |
| 阅读体验微调 | 推荐算法 |
| 种子内容 | 教程系列 |
| 轻量 yes/no 反馈 | 评论系统 |

**前置依赖：** v3.9（可靠的生成体验）+ 基础设施修复（埋点数据可用）

**主要风险：**
- 数据量太少导致无法得出结论（缓解：种子内容补充 + 手动测试）
- 优化方向可能是错的（缓解：小步快跑，每个改动可回滚）

**成功标准：**
- 产出获客漏斗报告，至少 1 个渠道被验证为有效
- /explore 页面教程点击率 >= 5%
- 至少 1 个自然搜索来源的访问
- 种子教程 >= 30 篇
- 轻量反馈日均参与 > 0

**总工作量：** 1-2 周

## 5. Phase 3：v3.11 — 创作者留存与反馈回路

**核心目标：给创作者正反馈，让他们愿意留。**

**为什么现在做这个：**
- v3.10 证明有人来了，现在需要让创作者有理由回来
- "发布后无反馈"是复访意愿最大的杀手（用户价值 Agent + 增长 Agent 共识）
- 创作者仪表盘是现有数据的新展示，不需要新功能

### 5.1 创作者仪表盘

**做什么：**

1. **基础阅读数据可见**
   - 利用已有 `events` 表聚合 `tutorial_viewed` 事件
   - 在已发布教程卡片上显示阅读量
   - 在 `/u/[username]` 档案页显示总阅读量、教程数
   - 新增简单仪表盘展示：总阅读量趋势（最近 7 天）、最受欢迎教程

2. **数据少时的处理**
   - 数据为零时展示鼓励性内容和创作引导，而不是空洞的零

### 5.2 基础互动

**做什么：**

1. **教程点赞**
   - 最轻量的互动，不需要复杂的内容审核
   - 点赞数据出现在创作者仪表盘
   - 教程卡片和探索页展示点赞数
   - 新增 `likes` 表 + API

### 5.3 通知基础

**做什么：**

1. **站内通知**
   - 阅读里程碑通知："你的教程被阅读了 50 次"
   - 点赞通知："有人赞了你的教程"
   - 在 AppShell 中加通知图标

2. **邮件通知（可选配置）**
   - 集成 Resend（可选配置，无配置则不发）
   - 阅读里程碑 + 周报摘要
   - 需要退订机制

### 5.4 草稿续写引导

**做什么：**
- 未完成草稿的高亮提醒
- "继续编辑"快捷入口
- 草稿列表按最近编辑排序

### 5.5 版本范围控制

| 包含 | 不包含 |
|------|--------|
| 创作者仪表盘（基础阅读数据） | 完整数据分析产品 |
| 教程点赞 | 评论系统 |
| 站内通知 + 可选邮件通知 | 实时推送 |
| 草稿续写引导 | 新手引导 overlay |
| | 用户关注/粉丝体系 |
| | 创作者收益/付费 |

**前置依赖：** v3.10（获客数据 + 种子内容带来流量）+ events 表数据积累

**对技术架构的要求：**
- 新增 `likes` 表（简单，符合当前模式）
- 邮件服务集成需要新依赖（Resend）
- 仪表盘可能需要引入轻量图表库

**主要风险：**
- 仪表盘数据量太少，展示效果不好（缓解：鼓励性内容 + 等数据积累）
- 邮件服务集成涉及域名验证（缓解：做成可选配置）

**成功标准：**
- 创作者仪表盘上线，展示真实数据
- 至少 1 个创作者因通知回访（通过埋点追踪）
- 邮件通知发出后 48 小时内回访率 >= 10%
- 创作者 7 天留存率（发布后 7 天内再次访问）有可测量提升

**总工作量：** 1-2 周

## 6. 明确不做什么（近期版本排除清单）

| 需求 | 排除原因 | 可能什么时候做 |
|------|---------|--------------|
| 实时协作编辑 | 需要 WebSocket + OT/CRDT，架构成本极高，数据模型不支持多写者 | v4.0 以后，且只做"异步协作"（分享编辑链接），不做实时 |
| 推荐算法 | 用户行为数据极薄，冷启动无解 | 积累 >= 1000 用户行为数据后 |
| 付费/商业化 | 商业模式未验证，用户基础不够 | 有 >= 100 活跃创作者后 |
| 教程系列/合集 | 需要 >= 30 篇教程和 >= 5 个活跃创作者 | v4.0 候选 |
| 评论系统 | 需要内容审核机制，用户量不够支撑活跃讨论 | v4.0 以后 |
| 移动端 App/PWA | 核心创作场景在桌面端 | 持续在每个版本做响应式改进，不集中做 |
| 国际化 UI (i18n) | 创作端已支持中英内容生成，UI 层 i18n 是锦上添花 | 有国际用户需求时 |
| 用户关注/社交图谱 | 用户密度不够 | 有 >= 100 活跃创作者后 |
| 嵌入式 JS SDK | 没有验证过外部集成需求 | 先做好 iframe embed（已有基础） |
| 团队/组织功能 | 没有验证过团队使用场景 | v5.0 以后 |
| 模板市场 | 需要 100+ 篇教程和活跃社区 | 远期 |
| 批量导入/管理 | 创作者数量少，单篇管理已够用 | 有明确需求后 |
