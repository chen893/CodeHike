## 一、逐页审查

### 1. 首页桌面端
1. **页面**：`/` + 桌面 1440x1100
2. **总评**：信息分区是清楚的，但首页像是“深色后台侧栏 + 白色卡片目录页”硬拼在一起，品牌气质和内容区层级没有真正打通。
3. **问题**
   1. 顶部 Hero 卡片体量很大，但内容只有标题、短描述和 3 个按钮，信息密度偏低，首屏利用率不高。
   2. 左侧深色侧栏占宽明显，右侧卡片被压窄，教程卡片标题和摘要都显得拥挤。
   3. “已发布教程”卡片样式重复度高，标题、标签、摘要、元信息的视觉权重接近，扫读效率一般。
   4. 底部“示例教程”卡片和上方真实教程卡片不是一套视觉系统，像另一个产品模块。
4. **优化建议**
   1. 缩短 Hero 高度，把标题区、筛选入口和核心 CTA 压到同一视觉带，释放更多首屏给教程列表。
   2. 收窄左侧侧栏或降低其存在感，让内容栅格更宽，至少保证教程卡片标题和摘要有更舒展的排版空间。
   3. 拉开教程卡片内部层级：标题更强、摘要更轻、元信息更收敛，减少每张卡片“哪里都在抢注意力”的问题。
   4. 统一“示例教程”和“已发布教程”的卡片框架、按钮风格和标签语言，避免首页出现两套组件语法。

2. **总评**：移动端能完整承接桌面内容，但首页变成连续长列表后，节奏过于单调，首屏品牌感也明显变弱。
3. **问题**
   1. 顶部 Hero 卡片被压得很紧，标题、说明和按钮都偏小，首屏没有拉开欢迎区与内容区的节奏差。
   2. 教程列表单列直排后重复感很强，卡片之间缺少更明确的分组或节奏变化，滚动疲劳明显。
   3. 每张卡片的按钮和元信息都很密，移动端一屏内同时出现太多细碎元素，阅读和点击都偏累。
   4. 页面底部“示例教程”模块孤零零地挂在列表末尾，重要性和前面内容不匹配。
4. **优化建议**
   1. 把 Hero 做成更明确的移动端首屏模块，减少按钮数量或改成主次两级 CTA，避免三按钮并排挤压。
   2. 用更强的分段标题、间隔或精选区，打破“同款卡片无限下落”的滚动体验。
   3. 放大卡片主标题与主按钮，收起次要元信息，让移动端决策点更聚焦。

   2. 空白代码编辑区高度很大，在默认态下形成一整块沉重的深色空窗，抢走了本应属于表单说明的注意力。

   2. 在默认空状态下压缩代码区高度，先给出导入提示和示例，再在用户开始编辑后展开。
   3. 统一表单组件的边框粗细、圆角和按钮语义，减少“工具式界面”的拼装感。
   4. 把主 CTA 做成更贴近表单结尾的固定底栏或明显结束区，缩短填写与提交之间的心理距离。

## 二、跨页总结

### 1. 重复出现的视觉问题
1. **深色壳层与浅色内容区割裂明显**：出现在首页桌面、创建页桌面、草稿列表桌面、教程页桌面、预览页桌面。左侧深色区域很强，但右侧内容系统没有承接同样的品牌语言。
2. **正文/卡片信息层级不够拉开**：出现在首页、草稿列表、教程阅读页、预览页。标题、摘要、标签、元信息经常同时抢注意力。
3. **移动端把桌面信息直接压缩**：出现在首页移动端、创建页移动端、草稿列表移动端、草稿编辑器移动端。信息虽然没丢，但明显缺少移动端再编排。
4. **大面积空白与大面积深色块同时存在**：出现在创建页、教程页、预览页、草稿编辑器。页面要么显空，要么显重，中间缺少平衡。

### 2. 设计系统不统一的地方
1. **颜色系统**：青蓝色、深蓝色、绿色、红色、浅灰边框在不同页面的语义并不稳定，主按钮有时是亮青，有时是深色，有时又退成浅描边。
2. **字体层级**：大标题有些页面非常强势，有些页面又依赖细小的辅助文字支撑说明，标题和说明之间的级差不稳定。
3. **间距系统**：教程阅读页和预览页间距偏松，列表页和编辑器页又偏密，缺少统一的“呼吸节奏”。
4. **组件风格**：教程卡片、示例卡片、列表卡片、预览工具条、表单按钮并不像同一家组件库产物。

### 3. 桌面端 vs 移动端差异最大的问题
1. 桌面端的主要问题是**左右分栏比例失衡**，尤其教程和预览页里代码区过重、正文区过窄。
2. 移动端的主要问题是**任务没有被重新切片**，桌面上的复杂信息只是简单下压成长页面，导致编辑和浏览都变得疲劳。
3. 同一页面在桌面端更像“后台工具”，在移动端则更像“设置长表单”，品牌感与沉浸感两头都没有站稳。

### 4. 用户体验最大的 3 个痛点
1. **创建和编辑流程太像填复杂配置面板**：尤其 `/new` 和 `/drafts/<draftId>`，用户很难在视觉上感知“我正在构建一篇教程”的成就路径。
2. **教程阅读页的版心分配不理想**：代码区长时间占据高权重，但正文区列宽偏紧，既影响阅读也削弱了代码变化的高光时刻。
3. **全站缺少统一的主次动作语义**：不同页面主按钮、次按钮、危险按钮风格不稳定，用户需要反复重新学习界面规则。

## 三、优先级排序

### P0（必须修复）
- `/drafts/<draftId>` 移动端：复杂编辑控件直接压缩到窄屏，patch 和工具栏可读性、可点击性都明显不足，已经接近功能不可用边缘。
- `/sample`、`/sample/request`、`/drafts/<draftId>/preview`、`/drafts/<draftId>/preview/request` 桌面端：正文栏长期过窄，影响教程主体内容的连续阅读，这是核心场景的直接体验损伤。
- `/drafts/<draftId>/preview/request` 桌面端：深色操作条与深色代码区叠加，工具层干扰正文，预览主任务被明显打断。

### P1（应该修复）
- `/`、`/drafts`、`/new` 桌面端：左侧深色壳层和右侧内容区的品牌风格脱节，首页与后台页整体气质不统一。
- `/`、`/drafts` 移动端：卡片列表密度过高，缺少清晰的扫描路径和节奏变化。
- `/new` 桌面端与移动端：默认空状态下的代码区过大，压缩了教学意图与提交动作的引导效率。
- `/drafts/<draftId>` 桌面端：首屏同时打开的编辑区过多，认知压力过高。

### P2（可以改善）
- 全站状态色与按钮语义进一步统一，减少“每个页面各自定义主按钮”的感觉。
- 教程页和预览页的章节留白可以再细调，增强滚动节奏。
- 首页“示例教程”模块和主列表卡片统一语言，让首页更完整。

## 四、修复建议

### 1. 重做教程/预览页的版心分配
- **涉及页面**：`/sample`、`/sample/request`、`/drafts/<draftId>/preview`、`/drafts/<draftId>/preview/request`
- **建议修改的内容**：收窄左侧代码舞台占比，放宽正文栏，压缩章节之间的垂直空白，并强化当前步骤与代码变化提示。
- **预期效果**：阅读更顺，scrollytelling 的“讲解 + 代码变化”关系更清晰，核心消费场景体验明显提升。

### 2. 把移动端从“压缩桌面”改成“重新编排任务”
- **涉及页面**：`/new`、`/drafts`、`/drafts/<draftId>`、`/sample`
- **建议修改的内容**：减少并列控件、增加分段标题和折叠层，给移动端建立更明确的任务流，一次只暴露当前需要处理的模块。
- **预期效果**：降低长页面疲劳和误触风险，提升创建、管理、编辑三类核心任务的移动端完成率。

### 3. 统一全站主次动作和状态色语义
- **涉及页面**：首页、创建页、草稿列表、编辑器、预览页
- **建议修改的内容**：定义稳定的主按钮、次按钮、危险按钮和状态标签颜色规则，并在所有卡片、表单、工具条里复用。
- **预期效果**：用户不需要反复学习按钮含义，界面整体会更像一个成熟产品，而不是多个阶段性页面的集合。

### 4. 减少“空大块”和“重深色块”的同时出现
- **涉及页面**：`/new`、`/sample`、`/drafts/<draftId>`、预览相关页面
- **建议修改的内容**：压缩默认空状态编辑器和大面积留白，同时降低深色区域长期占屏比例，让真正有信息的区域更靠前。
- **预期效果**：页面第一眼更聚焦，视觉压力下降，用户更容易理解当前该看哪里、该做什么。

### 5. 让创建与编辑流程更像内容创作，而不是配置后台
- **涉及页面**：`/new`、`/drafts/<draftId>`
- **建议修改的内容**：按步骤组织表单和编辑区，强化“当前阶段”“下一步动作”“完成状态”这三类提示，减少一次性暴露的高级设置。
- **预期效果**：创作过程更有方向感，新用户更容易上手，复杂能力也不会在默认态里压垮首屏。
