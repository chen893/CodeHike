# v3.6 Task Breakdown — Measure, Reliability, and Seed Growth

> **目标**：让产品可见（数据 + 测量 + 反馈循环），提升 AI 生成可靠性，建立读者转创作者的增长路径。

---

## 总览

### 分层原则

- `app/` → 调用 `lib/services/`，不直接调用 `lib/repositories/`、`lib/db/`、`lib/tutorial/`
- `lib/services/` → 调用 `lib/repositories/`、`lib/ai/`、`lib/monitoring/`、`lib/utils/`
- `lib/repositories/` → 调用 `lib/db/`
- `lib/monitoring/` → 纯工具层，不依赖 `lib/services/` 或 `lib/repositories/`
- `lib/ai/` → 不依赖 `lib/services/` 或 `lib/repositories/`

### 串行顺序与依赖图

```text
T1 (events table)        ─── DB schema + migration，所有 Track A 任务的前提
  │
  ├── T2 (analytics infra) ─── lib/monitoring/analytics.ts，依赖 T1 的 schema 类型
  │     │
  │     ├── T3 (core event helpers) ─── 3 个核心事件的 emit 函数
  │     │     │
  │     │     ├── T4 (view tracking) ─── app/[slug]/page.jsx 接入
  │     │     ├── T5 (creation tracking) ─── 创建流程接入
  │     │     └── T6 (CTA tracking) ─── CTA 组件接入
  │
T7 (source preprocessor) ─── lib/ai/source-preprocessor.ts，独立于 Track A
  │
  ├── T8 (patch auto-fix) ─── lib/ai/patch-auto-fix.ts，依赖 T7 提供的预处理上下文
  │     │
  │     ├── T9 (post-gen validation loop) ─── 修改 multi-phase-generator.ts，依赖 T8
  │     │     │
  │     │     └── T10 (reliability metrics dashboard) ─── admin 页面，依赖 T9 产出的指标
  │
T11 (reading-end CTA)    ─── 独立前端组件
  │
T12 (sitemap)            ─── 独立 Next.js 路由文件
  │
T13 (robots.txt)         ─── 独立 Next.js 路由文件
  │
T14 (export markdown)    ─── lib/services/ + API route
  │
T15 (export HTML)        ─── lib/services/ + API route，可与 T14 并行但推荐串行（共享模式）
```

**推荐执行顺序**：T1 → T2 → T3 → T7 → T8 → T4 → T5 → T6 → T9 → T11 → T12 → T13 → T14 → T15 → T10

理由：
- T1-T3 是基础设施，先铺好后续 Track A 的接入任务可以快速完成
- T7-T8 是 AI 可靠性的核心纯逻辑，与 Track A 无耦合，可以在 T3 之后立即开始
- T4-T6 是轻量接入点，可以在 T8 完成后穿插完成
- T10 是仪表盘，依赖 T9 产出的指标数据，放在最后
- Track C 的 T11-T15 都是独立的前端/路由任务，可以在 AI 逻辑之后逐步完成

## T1: Events 数据库表与迁移

**对应**：3.6.A2

**目标**：在数据库中新增 `events` 表，用于存储所有业务事件（page view、tutorial creation、CTA click 等）。

**修改文件**：
- `lib/db/schema.ts` — 新增 `events` 表定义
- `drizzle/` — 新增迁移 SQL 文件

**表设计**：
```text
events
  id          uuid PK defaultRandom
  eventType   varchar(64) NOT NULL        — 如 "tutorial_viewed", "tutorial_created", "cta_clicked"
  payload     jsonb DEFAULT '{}'          — 事件附加数据（slug, ctaId, source 等）
  userId      text                         — 可空，匿名用户为 null
  sessionId   varchar(128)                — 可空，用于匿名用户追踪
  slug        varchar(256)                — 可空，与教程关联的事件填写
  createdAt   timestamp with tz defaultNow
```

**分层**：
- schema 定义在 `lib/db/schema.ts`
- 迁移通过 `npx drizzle-kit generate` 生成

**完成标准**：
1. `lib/db/schema.ts` 中新增 `events` 表导出
2. `drizzle/` 目录下有对应迁移 SQL
3. `npx drizzle-kit push` 或手动执行迁移后表可正常创建
4. 现有测试 `npm test` 不回归

**测试**：无需新增测试（表定义层）

## T8: Patch Auto-Fix

**对应**：3.6.B1

**目标**：新增 patch 自动修复模块，在 AI 生成的 patch 匹配失败时尝试自动修复（空白对齐、缩进修正、部分匹配回退）。

**修改文件**：
- `lib/ai/patch-auto-fix.ts` — 新建

**依赖**：T7（使用预处理后的代码上下文进行智能匹配）

**设计要点**：
```typescript
// lib/ai/patch-auto-fix.ts
import type { ContentPatch } from '../schemas/tutorial-draft';
import type { PreprocessedSource } from './source-preprocessor';

export interface AutoFixResult {
  success: boolean;
  fixedPatches: ContentPatch[];
  fixesApplied: string[];   // 描述每次修复的字符串
}

export function tryAutoFixPatches(
  currentCode: Record<string, string>,
  patches: ContentPatch[],
  primaryFile: string,
  context?: PreprocessedSource[]
): AutoFixResult;
```

**修复策略**（按优先级尝试）：
1. 尾部空白/换行对齐
2. 缩进统一（tab ↔ spaces）
3. 模糊子串匹配（找到唯一近似匹配）
4. 上下文窗口匹配（扩大搜索范围到包含目标行的上下文块）

**分层**：
- `lib/ai/` 内部纯函数模块
- 被 `multi-phase-generator.ts` 在 patch 校验失败时调用

**完成标准**：
1. `tryAutoFixPatches` 在 patch 匹配失败时尝试修复
2. 修复成功返回 `success: true` 和 `fixedPatches`
3. 修复失败返回 `success: false`，不影响原流程
4. 每次 fix 记录到 `fixesApplied` 数组

**测试**：
- `tests/patch-auto-fix.test.js` — 覆盖：
  - 尾部空白修复
  - 缩进不一致修复
  - 模糊匹配修复
  - 无法修复时返回 success: false
  - 多文件 patch 的 primaryFile 回退

## T9: Post-Generation Validation Loop

**对应**：3.6.B2

**目标**：在多阶段生成器中，当 step 生成后 patch 校验失败时，自动调用 auto-fix 并在修复失败时重试该步生成（带错误上下文反馈给 AI）。

**修改文件**：
- `lib/ai/multi-phase-generator.ts` — 修改 step-fill 循环逻辑

**依赖**：T7, T8

**设计要点**：
1. 在现有的 `applyContentPatches` 校验之后，增加 auto-fix 逻辑
2. auto-fix 成功 → 使用修复后的 patch，继续
3. auto-fix 失败 → 作为 retry 的错误上下文，重新生成该 step
4. 最大 auto-fix 尝试次数 = 1（只尝试一次修复）
5. 重试总数受 `MAX_STEP_RETRIES` 约束（已有逻辑不变）

**修改范围**（multi-phase-generator.ts）：
```diff
+ import { tryAutoFixPatches } from './patch-auto-fix';
+ import { preprocessSources, type PreprocessedSource } from './source-preprocessor';

  // 在 createMultiPhaseGenerationStream 函数内部：
+ // 预处理 source items
+ const preprocessed = preprocessSources(sourceItems);

  // 在 step 生成的 retry 循环中，patch 校验位置：
- if (step.patches && step.patches.length > 0) {
-   applyContentPatches(previousFiles, step.patches, primaryFile);
- }
+ if (step.patches && step.patches.length > 0) {
+   try {
+     applyContentPatches(previousFiles, step.patches, primaryFile);
+   } catch (patchErr) {
+     // 尝试 auto-fix
+     const fixResult = tryAutoFixPatches(previousFiles, step.patches, primaryFile, preprocessed);
+     if (fixResult.success) {
+       step.patches = fixResult.fixedPatches;
+       console.log(`[multi-phase] Auto-fix applied for step ${i + 1}:`, fixResult.fixesApplied);
+     } else {
+       // auto-fix 失败，抛出错误触发 retry
+       throw new Error(`Patch match failed, auto-fix unable to resolve: ${patchErr.message}`);
+     }
+   }
+ }
```

**完成标准**：
1. Patch 匹配失败时，先尝试 auto-fix 再决定是否 retry
2. Auto-fix 成功时使用修复后的 patch
3. Auto-fix 失败时将错误信息传递给 AI 作为 retry 上下文
4. SSE 流中不新增 event type（对前端透明）
5. 现有测试 `npm test` 不回归

**测试**：
- `tests/patch-chain.test.js` — 如果有覆盖 multi-phase 逻辑的测试需确认通过
- 考虑新增 `tests/validation-loop.test.js` 测试 auto-fix 集成的纯逻辑路径

## 风险与注意事项

1. **T1 迁移**：需要在有 DATABASE_URL 的环境中执行 `drizzle-kit generate`。迁移文件命名遵循 drizzle 惯例（timestamp + snake_case 描述）。
2. **T2 analytics 静默失败**：所有 `trackEvent` 调用必须 try-catch 并 console.error，绝对不能让 analytics 失败影响主业务流程。
3. **T9 multi-phase-generator 修改**：这是核心 AI 生成链路（262 行），修改范围需要控制在 step-fill 循环的 patch 校验段，不改变整体 SSE 流结构。修改前确保现有 `tests/patch-chain.test.js` 全部通过。
4. **T6 客户端事件追踪**：需要考虑 CSRF 和速率限制，防止 events 表被恶意刷入。初始版本可仅做基本校验（eventType 白名单）。
5. **T10 admin 页面鉴权**：初始版本可以先不做 admin 鉴权（内部工具），后续通过 NextAuth session 检查限制访问。
6. **分层遵守**：所有新增代码遵循 `app/ → components/ → lib/services/ → lib/repositories/` 的分层规则，不在 `app/` 直接调用 `lib/repositories/` 或 `lib/db/`。
