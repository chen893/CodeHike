## 3. Newly Discovered Issues

1. **Patch auto-fix is conservative** — Only handles whitespace and tab/space issues. Does NOT handle AI hallucinating code that doesn't exist at all. This is the harder problem and requires prompt engineering, not string matching.

2. **Events table will grow unbounded** — No TTL or cleanup mechanism. At low volume (<1000 events/day) this is fine. Need a cleanup job before v3.8.

3. **Export services re-run the assembler pipeline** — `export-markdown.ts` and `export-html.ts` call `applyContentPatches` independently. This duplicates work that the assembler already does. For large tutorials this could be slow. A cached approach would be better for v3.7.

4. **Sitemap uses `getHomePageData()` which loads full tutorial snapshots** — For sitemap generation, we only need slugs and dates. Should have a lightweight `listAllPublishedSlugs()` query.

5. **Source preprocessor structure extraction is regex-based** — For complex languages (JSX, TypeScript generics, Python decorators) the regex patterns may miss or incorrectly capture structures. An AST-based approach would be more reliable but adds a heavy dependency.

## 5. Lessons Learned

### What worked well
- **Review-fix-verify loop** caught 3 real issues (infinite loop, duplicate constants, fetch abort). Worth the overhead.
- **Parallel agent execution** (3 agents simultaneously) dramatically speed up the analysis phase.
- **Shared constants pattern** — Extracting `event-types.ts` early prevents future sync bugs.

### What could improve
- **Source preprocessor integration was left incomplete** — Building it without integrating into the actual AI pipeline means it's dead code until someone connects it. Should have done the integration as part of the same task.
- **Admin dashboard (T10) should have been explicitly scoped out** — Rather than "deferred", it should have been marked "not in v3.6" from the start.
- **Sitemap efficiency** — Using `getHomePageData()` was expedient but wasteful. A dedicated query would be better engineering.

### Rules for future iterations
1. If a module is built, integrate it into the pipeline in the same task. No orphaned modules.
2. Test coverage must include the integration point, not just the pure function.
3. When the task list says "optional context", treat it as "future debt" and flag it.
