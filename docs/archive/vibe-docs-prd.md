# 产品需求文档（PRD）：VibeDocs v3.0

**文档版本：** v3.0  
**最后更新：** 2026-04-08  
**产品定位：** AI 驱动的源码教学内容生成器  
**一句话定义：** 用户输入一段源码和明确的教学意图，系统生成一份可编辑、可预览、可发布的逐步构建式教程草稿。

---

## 0. 本版原则

本版 PRD 以当前系统已经跑通的“数据 -> 组装 -> 页面渲染”链路为唯一技术基线，不再沿用旧版中与现状不一致的假设。

这意味着：

1. 系统的核心资产是结构化教程数据，而不是 HTML 页面。
2. 当前渲染层已经足以支撑“教程草稿 -> 预览页面”的流程，后续产品能力应建立在这条链路之上。
3. 系统的顶层持久化对象应为 `DraftRecord`，其中 `tutorialDraft` 是权威教程内容；不能再为不同阶段定义不同页面模型。
4. 未来即使接入真实后端、编辑器和发布能力，当前的教程数据 DSL、组装层和渲染器仍应保持为稳定底座。

---

## 1. 产品定义

VibeDocs 的目标，是把“读源码 -> 提炼教学主线 -> 组织步骤 -> 生成讲解 -> 编辑发布”这条链路产品化。

它不是一个代码高亮工具、Diff 工具或录屏工具的增强版，而是一个面向技术作者的教学内容生产系统。系统的核心产物不是页面 HTML，也不是单次模型输出，而是一份可持续编辑和发布的结构化教程数据。

VibeDocs 要解决的问题不是“让代码看起来更炫”，而是“把源码讲明白，并把讲明白这件事沉淀成可发布内容”。

---

## 2. 当前系统基线

当前仓库已经具备一条完整、可运行的教程渲染链路。它还不是完整产品，但已经验证了最重要的一件事：

> 只要有一份符合约定的数据结构，系统就能稳定地把它组装为可预览、可访问的教学页面。

### 2.1 当前已实现能力

当前系统已经实现：

- 通用教程数据 DSL
- 服务端组装层
- 通用 CodeHike 渲染器
- 静态教程页
- 远程请求教程页
- 教程 payload API
- 一份中性的 sample 教程数据

### 2.2 当前已实现路由

| 路由 | 作用 |
|------|------|
| `/` | 通用首页，说明系统和可用样例 |
| `/[slug]` | 服务端直出教程页 |
| `/[slug]/request` | 客户端请求 payload 后渲染教程页 |
| `/api/tutorials/[slug]` | 返回前端可直接消费的教程 payload |

当前默认样例：

- `/sample`
- `/sample/request`
- `/api/tutorials/sample`

### 2.3 当前关键实现文件

| 文件 | 职责 |
|------|------|
| `content/sample-tutorial.js` | 示例教程数据 |
| `lib/tutorial-registry.js` | `slug -> tutorial` 注册表 |
| `lib/tutorial-assembler.js` | 把教程 DSL 组装成高亮步骤 |
| `lib/tutorial-payload.js` | 把教程 DSL 转成前端 payload |
| `components/tutorial-scrolly-demo.jsx` | 通用教程渲染器 |
| `components/remote-tutorial-page.jsx` | 远程请求页容器 |
| `app/[slug]/page.jsx` | 静态教程页 |
| `app/[slug]/request/page.jsx` | 远程教程页 |
| `app/api/tutorials/[slug]/route.js` | 教程 payload API |

### 2.4 当前未实现能力

当前系统尚未实现，但属于产品下一阶段必须补齐的能力：

- 源码输入界面
- 教学意图输入界面
- AI 生成教程草稿
- 草稿编辑器
- 草稿版本管理
- 发布后台与持久化存储

因此，v3.0 的核心任务不是重写渲染层，而是在当前渲染底座之上补齐“输入 -> 草稿 -> 编辑 -> 预览 -> 发布”的产品链路。

## 3. 问题与机会

技术作者在解释源码时，通常会遇到四个连续问题：

1. 看懂源码不等于能讲清源码。
2. 即使知道讲什么，也很难组织出适合发布的逐步构建结构。
3. 现有工具更擅长展示代码，不擅长沉淀成可编辑的教学草稿。
4. 生成结果如果不能进入稳定预览和发布链路，就无法成为真正的内容生产系统。

当前系统已经证明：

- 结构化教程数据可以被稳定渲染
- 同一份教程数据可以同时支持静态预览和远程预览
- 渲染层不需要重新理解教学结构，只需要消费标准数据

这给 VibeDocs 提供了一个明确的产品机会：

> 把 AI 的输出目标从“直接写页面”改成“生成结构化教程草稿”，然后复用当前稳定的渲染层完成预览与发布。

---

## 4. 产品目标与非目标

### 4.1 v3.0 产品目标

v3.0 的目标是把当前“教程渲染应用”推进为“源码教学内容生成器”的第一版闭环。

具体目标：

1. 用户能够输入源码和教学意图。
2. 系统能够生成符合当前 DSL 的教程草稿。
3. 用户能够在草稿上继续编辑，而不是只能接受一次性结果。
4. 用户能够立即用当前渲染器预览结果。
5. 用户能够将同一份草稿发布为稳定可访问内容。

### 4.2 v3.0 成功标准

v3.0 成功不以动效数量或模型复杂度衡量，而以以下结果衡量：

- AI 生成结果能进入现有渲染链路，无需人工二次转格式。
- Creator 愿意在生成结果上继续编辑，而不是推翻重来。
- 预览结果与发布结果基于同一份教程草稿，没有语义漂移。
- 单篇教程能在一次会话内从输入源码推进到可发布状态。

### 4.3 非目标

v3.0 不追求以下能力：

- 做成通用博客 CMS
- 做成任意仓库的全自动课程工厂
- 构建社区、推荐、评论、关注体系
- 构建视频录制或动图生产系统
- 构建大型代码运行沙箱平台
- 在渲染层引入第二套与当前 DSL 不兼容的数据模型

## 5. 用户角色

### 5.1 核心用户：Creator

Creator 是 v3.0 的唯一核心设计对象，典型角色包括：

- 技术博主
- 课程讲师
- 开源维护者
- 团队内部技术分享作者

Creator 的核心诉求不是“看代码”，而是“把源码讲清楚并形成可发布内容”。

### 5.2 次级用户：Reader

Reader 是已发布教程的阅读者。Reader 不是 v3.0 的主要付费或运营对象，但 Reader 是否能读懂内容，直接决定 Creator 是否愿意发布。

## 6. 核心概念与统一术语

为了避免旧版 PRD 里术语漂移，本版统一如下：

| 术语 | 定义 |
|------|------|
| `Source Item` | 一个输入源码项，可以是粘贴片段，也可以是外部文件 |
| `Teaching Brief` | 教学意图，定义讲什么、给谁讲、重点与边界 |
| `DraftRecord` | 系统内部的顶层草稿记录，承载输入、教学意图、教程内容与状态 |
| `Tutorial Draft` | `DraftRecord` 内部的权威教程内容结构，也是编辑器直接操作的内容对象 |
| `Tutorial Payload` | 由服务端组装后，前端可直接渲染的预览数据 |
| `Step` | 教程中的一个独立教学单元 |
| `Patch` | 某一步相对上一步的代码增量描述 |
| `Preview` | 基于 `Tutorial Draft` 或 `Tutorial Payload` 的页面渲染结果 |
| `Published Tutorial` | 一个已发布、可稳定访问的教程快照 |

术语使用规则：

- 持久化阶段统一围绕 `DraftRecord`
- 教程内容生成与编辑阶段统一围绕 `DraftRecord.tutorialDraft`
- 页面渲染阶段统一围绕 `Tutorial Payload`
- 不再使用“页面数据”“文章数据”“教学包”“内容包”等含糊概念

## 7. 核心数据模型

### 7.1 DraftRecord：系统的顶层持久化对象

v3.0 接受一个上层 `DraftRecord` 作为唯一持久化对象。它负责把源码输入、教学意图、教程内容和状态放进同一条记录中：

```js
{
  id,
  sourceItems: [
    {
      id,
      label,
      content,
    },
  ],
  teachingBrief: {
    topic,
    audience_level,
    core_question,
    ignore_scope,
    output_language,
  },
  tutorialDraft: { ... },
  status: "draft" | "published",
  createdAt,
  updatedAt,
}
```

它的意义是：

- 让输入、生成、编辑、预览、发布都能追溯到同一条记录
- 避免为了补教学意图或源码来源，再额外发明一层包裹对象
- 为后续远程预览、保存草稿、发布快照提供稳定主键

### 7.2 Tutorial Draft：DraftRecord 内部的权威教程内容结构

当前系统已经验证，以下结构足以支撑页面渲染，因此 v3.0 默认把它定义为 `DraftRecord` 内部的权威教程内容结构：

```js
{
  meta: {
    title,
    lang,
    fileName,
    description,
  },
  intro: {
    paragraphs: [],
  },
  baseCode: `...`,
  steps: [
    {
      id,
      eyebrow,
      title,
      lead,
      paragraphs,
      patches,
      focus,
      marks,
    },
  ],
}
```

它的意义是：

- 既能表达教学结构
- 又能表达代码变化
- 还能直接进入当前组装层

因此它不是“仅用于渲染的临时格式”，而应作为：

- AI 生成的核心内容格式
- 编辑器直接操作的内容格式
- 预览与发布时被冻结或派生的源内容格式

### 7.3 Tutorial Payload：渲染层消费格式

当前远程页和 API 已经证明，前端真正需要消费的是一层组装后的 payload：

```js
{
  title,
  description,
  fileName,
  intro: [],
  steps: [
    {
      eyebrow,
      title,
      lead,
      paragraphs,
      highlighted,
    },
  ],
}
```

它是由服务端从 `Tutorial Draft` 派生出来的。

结论：

- `Tutorial Draft` 是源数据
- `Tutorial Payload` 是渲染数据
- 两者必须有明确单向关系，不能在前端反向推导

### 7.4 Teaching Brief：生成质量的上游约束

v3.0 明确要求，源码输入不是唯一输入，必须同时提供 `Teaching Brief`。

P0 必填字段：

- `topic`
- `audience_level`
- `core_question`
- `ignore_scope`
- `output_language`

P1 可选字段：

- `desired_depth`
- `target_step_count`
- `preferred_style`

原则：

- 没有明确教学意图时，不应直接开始生成
- 同一段源码在不同 `Teaching Brief` 下，应生成不同教程草稿

### 7.5 Published Tutorial：发布后的冻结快照

“发布”不是给草稿开一个可分享预览链接，而是从 `DraftRecord` 冻结出一份独立的 `Published Tutorial`：

```js
{
  id,
  draftRecordId,
  slug,
  tutorialDraftSnapshot: { ... },
  publishedAt,
}
```

它的意义是：

- 发布结果拥有独立的公开身份和稳定 URL
- 后续继续编辑 `DraftRecord` 时，不会隐式污染已发布内容
- 发布页可以继续沿用当前 payload 组装链路，而不需要第二套页面模型

## 8. 当前渲染方案：v3.0 的固定底座

v3.0 不重做渲染层，而是把当前渲染方案视为固定底座。

### 8.1 已验证的链路

当前真实链路是：

```text
DraftRecord
  -> tutorialDraft
  -> buildTutorialSteps()
  -> TutorialScrollyDemo
  -> 页面渲染
```

远程链路则是：

```text
DraftRecord
  -> tutorialDraft
  -> buildTutorialPayload()
  -> /api/tutorials/[slug]
  -> RemoteTutorialPage
  -> TutorialScrollyDemo
```

### 8.2 当前渲染层的关键约束

当前渲染器 `components/tutorial-scrolly-demo.jsx` 有几条必须保留的约束：

1. 左侧固定代码舞台，右侧独立滚动文章区。
2. 代码区必须是同一个面板持续变化，不切整块组件。
3. 代码变化由同一个 `Pre` 实例承载，以保留 CodeHike token 过渡和增删行动画。
4. 浏览器端不做 patch 计算和高亮，相关工作全部在服务端完成。
5. 渲染器只消费标准化数据，不参与教学结构推断。

这意味着未来不管是 AI 生成结果、人工编辑结果还是发布结果，都必须服从当前渲染契约。

## 9. 端到端产品流程

### 9.1 v3.0 目标流程

```text
输入源码
  +
填写 Teaching Brief
  ->
创建 / 更新 DraftRecord
  ->
生成 tutorialDraft
  ->
进入草稿编辑界面
  ->
调用当前渲染链路进行预览
  ->
继续编辑 / 局部重生成
  ->
发布为稳定教程
```

### 9.2 当前系统已覆盖的部分

当前系统已经覆盖：

- `DraftRecord.tutorialDraft -> buildTutorialSteps() -> Preview`
- `DraftRecord.tutorialDraft -> Tutorial Payload -> Remote Preview`

### 9.3 v3.0 需要新增的部分

v3.0 需要新增：

- `Source Input + Teaching Brief -> DraftRecord`
- `DraftRecord -> tutorialDraft generation`
- `DraftRecord -> Editor`
- `DraftRecord -> Published Tutorial`

也就是说，v3.0 的重点不在“换渲染器”，而在“补上渲染器之前和之后的产品能力”。

## 10. 功能需求

### 10.1 P0：源码输入

系统必须支持至少一种明确的源码输入方式：

- 代码片段粘贴

P1 可扩展：

- GitHub 文件导入
- 多文件组合输入

要求：

- 输入不能被静默截断
- 每个 `Source Item` 必须有稳定标识
- 后续生成出的步骤必须能追溯到输入源码

### 10.2 P0：Teaching Brief 输入

系统必须要求用户明确教学意图，而不是“扔一段代码直接生成”。

要求：

- 必填字段缺失时，不允许开始生成
- 系统可以给建议值，但用户必须确认
- `Teaching Brief` 必须进入后续生成上下文

### 10.3 P0：AI 生成 Tutorial Draft

AI 生成模块的最低要求不是“写一篇像博客的文章”，而是在 `DraftRecord` 内生成一份符合当前 DSL 的 `tutorialDraft`。

最低要求：

- 有稳定标题和简介
- 有清晰步骤序列
- 每一步有讲解文本
- 每一步的代码变化可回放
- 结构能够直接进入当前 `buildTutorialSteps()`

关键约束：

- 生成结果必须可校验
- 不能生成无法被 `patch` 机制应用的数据
- 不能生成依赖前端猜测行号的结果

### 10.4 P0：草稿编辑

系统必须允许用户在生成结果上继续编辑。

P0 编辑能力应覆盖：

- 修改标题
- 修改简介
- 修改步骤标题与文案
- 新增步骤
- 重新生成单个步骤文案或结构

P1 再考虑：

- 删除步骤
- 调整步骤顺序
- 直接编辑 `patches`
- 直接编辑 `focus`
- 直接编辑 `marks`
- 可视化代码区区域选择

原则：

- 可编辑性优先于一次生成完美
- 局部修复优先于整篇重做

### 10.5 P0：预览

预览必须直接复用当前渲染链路，而不是再做一套页面系统。

v3.0 至少需要支持两类预览：

- 静态预览
- 远程预览

静态预览适用于：

- 本地草稿即时查看
- 快速检查结构与内容

远程预览适用于：

- 验证数据 API
- 未来接真实后端
- 发布前检查远程链路

补充说明：

- 远程预览链接服务于草稿检查，不等于发布链接
- 发布一定对应独立的 `Published Tutorial`

关键要求：

- 预览和最终发布必须共享同一套渲染器
- 不能出现“编辑页一套样子、发布页另一套样子”

### 10.6 P0：发布

发布的本质不是导出 HTML，也不是简单生成一个“可分享的草稿预览链接”，而是从 `DraftRecord` 冻结出一份独立的 `Published Tutorial` 快照，并赋予稳定公开入口。

最低要求：

- 发布动作基于当前 `DraftRecord`
- 发布结果持有独立的 `tutorialDraftSnapshot`
- 发布后有稳定 slug
- 发布结果可以直接进入当前预览渲染链路
- 发布快照不可被后续草稿编辑隐式污染

P1 再考虑：

- 草稿版本历史
- 发布回滚
- 多 revision 管理

## 11. 信息架构与页面形态

v3.0 至少需要以下四类产品页面：

| 页面 | 目标 |
|------|------|
| 输入页 | 输入源码与 Teaching Brief |
| 草稿工作区 | 编辑 `DraftRecord` 内的 `tutorialDraft` |
| 预览页 | 调用当前渲染链路检查当前草稿效果 |
| 发布页 | 对外展示 `Published Tutorial` 快照 |

当前仓库已经具备“预览页/发布页”的雏形，v3.0 的产品工作重点是补齐“输入页”和“草稿工作区”。

## 12. 技术原则

### 12.1 只维护一套权威教程内容格式

当前 DSL 已经验证可行，因此 v3.0 不允许再引入第二套与之并行的教程内容格式。`DraftRecord` 和 `Published Tutorial` 只是包裹层，不是并行内容模型。

### 12.2 服务端负责组装，前端负责消费

`patch` 应用、focus 注入、mark 注入、CodeHike 高亮都必须在服务端完成。前端只消费结果，不参与推导。

### 12.3 渲染层不重新理解教学结构

渲染层的职责是：

- 展示步骤
- 切换代码
- 表达变化

它不负责：

- 分析源码
- 猜测教学主线
- 自动修正文案结构

### 12.4 预览和发布必须同构

同一份草稿数据在预览和发布阶段必须沿用同一套渲染契约，减少内容漂移。

### 12.5 局部重生成必须优先于全量重做

这是产品层要求，也是系统层要求。否则 Creator 无法把 AI 结果当作真正的内容生产起点。

### Phase 3：发布闭环

### Phase 4：增强能力

## 14. 成功指标

v3.0 建议关注以下指标：

### 14.1 生成可用性

- 生成结果能直接进入当前渲染链路的比例
- `patch` 校验通过率
- 生成后无需手工修复结构错误的比例

### 14.2 创作效率

- 从输入源码到首次可预览结果的时间
- Creator 从生成到发布的完成率
- 单篇教程平均编辑轮次

### 14.3 内容质量

- Creator 对结果“可继续编辑”的主观评分
- Creator 对“教学主线是否清晰”的主观评分
- Reader 对“是否能读懂主线”的反馈

### 14.4 系统一致性

- 预览与发布一致率
- 静态预览与远程预览一致率

## 15. 风险与关键决策

### 15.1 风险：生成结果不符合当前 DSL

如果 AI 输出不稳定，系统将无法进入当前渲染链路。  
决策：把“生成符合 DSL 的草稿”定义为首要约束，而不是事后修补。

### 15.2 风险：步骤过粗，失去教学价值

如果每一步改动过大，渲染层虽然能展示，但教学效果会显著下降。  
决策：步骤粒度是产品质量问题，不只是文案问题。

### 15.3 风险：编辑器引入第二套数据模型

如果编辑器操作的不是 `DraftRecord.tutorialDraft`，系统会再次回到“生成一套、渲染一套”的割裂状态。  
决策：编辑器只能直接修改 `DraftRecord` 中的 `tutorialDraft`。

### 15.4 风险：发布流程绕开当前渲染层

如果发布页另建一套页面模型，预览与发布就会再次分裂。  
决策：发布结果必须继续复用当前渲染链路。

## 16. 最终结论

VibeDocs v3.0 的核心不是再设计一套“看起来更完整”的理想架构，而是在当前已跑通的系统基础上，把它推进为真正的产品闭环。

当前系统已经证明：

- 一份标准化教程数据可以稳定渲染为教学页面
- 同一份数据可以支持静态预览和远程预览
- 渲染层已经具备作为产品底座的稳定性

因此，v3.0 的正确方向是：

> 让 AI 在 `DraftRecord` 中生成 `tutorialDraft`，让 Creator 编辑它，让系统用当前链路预览它，并在发布时冻结为 `Published Tutorial`。

如果未来的任何设计偏离这条主线，例如：

- 让 AI 直接写页面而不是写草稿
- 让编辑器维护另一套数据结构
- 让发布页绕开当前渲染链路

都应视为偏离产品方向。
