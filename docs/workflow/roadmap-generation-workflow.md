你要基于我提供的和 /Users/chen/Downloads/project-copy/Apr/CodeHike/docs/tutorial-data-format.md 和 /Users/chen/Downloads/project-copy/Apr/CodeHike/docs/vibedocs-technical-handbook.md ，自动创建并调用多个子 agent，协同完成“后续版本路线图规划”。

目标不是重写 PRD，也不是只做点评，而是：
1. 理解当前系统已经做到哪里
2. 理解现有技术架构的能力边界与演进空间
3. 找出后续版本最值得推进的方向
4. 形成一个分阶段、可执行、有优先级的产品路线图

工作方式要求：
- 你必须使用多 agent 协作
- 每个子 agent 只看必要上下文，尽量隔离，避免互相影响判断
- 每个子 agent 先独立输出，再由主 agent 汇总
- 不要让所有 agent 输出风格趋同
- 每个子 agent 只回答自己职责范围内的问题，不越界替别的 agent 下结论
- 最终由主 agent 进行冲突收敛、优先级排序、版本切分

你需要创建以下子 agent，并让它们分别完成自己的职责：

---

【子 Agent 1：现状解析 Agent】
职责：
- 阅读现有 PRD 和当前技术架构，提炼“当前系统现状”
- 明确产品当前解决的问题、目标用户、核心流程、关键模块
- 识别当前版本已经具备的能力、尚未覆盖的能力、描述模糊的能力
- 输出一份“当前产品能力地图”，作为后续路线图讨论的基线
- 不负责提出大量新想法，重点是把现状讲清楚

输出重点：
- 当前产品定位
- 当前核心用户
- 当前核心业务流程
- 当前已上线/已具备能力
- 当前明显缺口
- 当前版本最重要的约束前提

---

【子 Agent 2：架构约束与演进 Agent】
职责：
- 只从技术架构角度分析当前系统
- 判断现有架构最适合支撑什么类型的后续演进
- 识别架构瓶颈、扩展风险、复杂度来源、未来可能拖慢迭代的地方
- 判断哪些产品方向与当前架构天然匹配，哪些方向会导致系统失控
- 给出“技术上可渐进演进”的路线，而不是理想化重构

输出重点：
- 当前架构适合承接的产品方向
- 当前架构的主要瓶颈
- 哪些需求适合继续叠加
- 哪些需求需要先补基础设施再做
- 哪些方向会显著增加系统复杂度
- 对路线图的技术约束建议

---

【子 Agent 3：用户价值与场景 Agent】
职责：
- 从用户问题和使用场景出发，评估当前系统的价值闭环
- 找出用户从进入产品到获得价值的关键路径
- 识别当前流程里的阻塞点、断点、低价值模块和伪需求
- 推导后续版本最应该补强的用户价值，而不是单纯堆功能
- 优先思考“用户为什么会持续使用”，而不是“还能加什么功能”

输出重点：
- 当前产品的核心价值主线
- 用户完成关键任务时的阻碍
- 最缺失的高价值能力
- 哪些现有模块价值不高或容易分散注意力
- 后续版本应该优先强化的用户收益

---

【子 Agent 4：增长与反馈闭环 Agent】
职责：
- 从增长、留存、激活、转化、复访角度分析当前产品
- 识别产品现阶段最需要建立的反馈闭环
- 判断路线图中哪些能力能提高获客、激活、留存、传播或商业化
- 强调可验证性，提出适合进入路线图的可测量方向
- 不要泛泛谈增长，要结合现有产品阶段提出合理建议

输出重点：
- 当前产品最缺的增长环节
- 最弱的用户转化/留存链路
- 哪些后续版本方向更容易产生业务增量
- 哪些建议可以被埋点或实验验证
- 对版本路线图的增长优先级建议

---

【子 Agent 5：产品策略与版本切分 Agent】
职责：
- 从产品经理视角，把“很多可能要做的事”整理成“应该怎么分版本做”
- 判断哪些内容适合放到下一版本，哪些适合延后
- 识别会导致版本目标失焦的内容
- 控制单个版本范围，避免版本膨胀
- 强调“每个版本只解决一类核心问题”

输出重点：
- 路线图应该按什么主线切分版本
- VNext、V+2、V+3 各自最合适的目标
- 每个版本该解决的核心矛盾
- 哪些需求不该进入近期版本
- 每个版本成功的判定标准

---

【子 Agent 6：风险与依赖 Agent】
职责：
- 识别路线图中的产品风险、技术风险、组织风险、协作风险和依赖关系
- 分析哪些功能看起来重要，但实施成本高、依赖多、失败概率大
- 判断路线图中的顺序是否合理，是否存在必须先做的前置项
- 帮助主 agent 避免输出“逻辑上漂亮但无法落地”的路线图

输出重点：
- 高风险方向
- 高依赖方向
- 必须前置建设的能力
- 容易低估工作量的模块
- 对版本排序的风险修正建议

---

【子 Agent 7：路线图汇总 Agent】
职责：
- 不直接提出全新观点，主要负责整合前面所有子 agent 的输出
- 识别共识点、冲突点和不确定点
- 在冲突中做 trade-off，而不是简单拼接所有建议
- 最终产出一版结构清晰、优先级明确的产品路线图
- 路线图必须同时考虑：用户价值、业务增长、技术可行性、版本节奏、实施风险

输出重点：
- 当前系统处于什么阶段
- 后续路线图主线是什么
- 为什么这样排序
- 每个版本的目标、核心功能、前置依赖、预期收益、风险提示
- 明确列出“不做什么”

---

执行要求：
1. 先让每个子 agent 独立输出，不要提前互相参考结论
2. 主 agent 在汇总时要先做“共识提炼”，再做“冲突裁决”
3. 最终路线图至少分为三个阶段：近期版本、中期版本、后续版本
4. 每个阶段都要回答：
   - 这一阶段的核心目标是什么
   - 为什么现在做这个
   - 要做哪些关键能力
   - 不做哪些能力
   - 对技术架构有什么要求
   - 有哪些主要风险
5. 不要输出空泛的大词，要尽量贴合我提供的系统现状
6. 不要默认推翻现有系统，要以“在现有基础上演进”为优先策略
7. 输出时优先形成“路线图”，不是长篇理论分析

---

输出文件命名规范：
1. 每次执行先确定 `run_id`，格式为 `<YYYYMMDD>-roadmap-<scope>`：
   - `YYYYMMDD` 使用执行当天日期
   - `scope` 使用小写 kebab-case，描述路线图范围；默认用 `post-current`
   - 示例：`20260414-roadmap-post-current`
2. 最终路线图固定写入 `docs/<run_id>.md`，例如 `docs/20260414-roadmap-post-current.md`。这个文件是对外阅读的主产物，必须放在 `docs/` 根目录。
3. 子 agent 中间稿和过程记录统一写入 `docs/roadmaps/<run_id>/`，不要散落到 `docs/` 根目录。
4. 子 agent 独立输出文件使用两位序号 + 职责 slug：
   - `01-current-state.md`
   - `02-architecture-evolution.md`
   - `03-user-value-scenarios.md`
   - `04-growth-feedback-loop.md`
   - `05-product-versioning.md`
   - `06-risks-dependencies.md`
   - `07-roadmap-synthesis.md`
5. 若需要记录汇总前的共识、冲突和裁决过程，写入 `docs/roadmaps/<run_id>/decision-log.md`；不要混进子 agent 文件或最终路线图。
6. 不要使用 `final.md`、`roadmap.md`、`new-roadmap.md`、`post-roadmap.md` 等临时名；最终文件名必须自带日期、roadmap 类型和范围。
7. 文件名全部小写，单词用 `-` 连接，只使用 ASCII 字母、数字和连字符；不使用空格、中文、括号、大小写混合或 `copy` / `final2` 等迭代后缀。
8. 如果同一天同范围重复执行，在 `run_id` 后追加 `-r2`、`-r3`，例如 `20260414-roadmap-post-current-r2`；不要覆盖旧结果。
