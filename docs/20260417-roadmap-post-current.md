# VibeDocs 路线图：从"功能完备"到"有人用、有人留"

> 基于 6 个子 Agent 的独立分析（现状解析、架构约束、用户价值、增长闭环、版本策略、风险依赖）整合而成。裁决原则：代码实际 > 产品直觉，技术可行性 > 功能完整性，风险规避 > 机会追求。

---

## 1. 当前系统处于什么阶段

**一句话：VibeDocs 是一个功能链路完整但核心价值尚未被验证的产品。**

具体判断：

- **创作闭环已跑通：** 源码输入 -> AI 三阶段生成 -> 编辑工作区 -> 预览 -> 发布 -> 公开访问，全链路可用
- **渲染引擎成熟度高：** scrollytelling 交互效果（左代码右文章、focus 高亮、行标记、变更指示器）是产品最直观的差异化竞争力
- **核心功能不可靠：** AI 生成成功率估计在 50-60%（无精确基线），多文件和渐进式快照的 source coverage 崩塌是系统性问题
- **零用户数据：** 6 个关键事件已注册但从未被调用，创作者反馈回路为零，产品在"黑暗中运行"
- **基础设施有债：** Drizzle migration 不完整（问题 32），关键路径测试覆盖不足，部署需手工补 SQL

**核心矛盾：** 产品功能完备但没有人用。不是缺功能，而是"核心价值（AI 生成的教程）还不够可靠"加上"首次体验门槛过高"加上"创作者没有回来的理由"。

## 2. 后续路线图主线

**从"功能完备"到"有人用"到"有人留"再到"数据验证"。**

```
v3.10 前置  基础设施修复（3-5 天）
    │  点亮数据 + 修好部署 + 补关键测试
    ▼
v3.10  首次体验破冰 + 阅读量可见化（1 周）
    │  60 秒出成品 + 创作者看到阅读数据
    ▼ 判定：快速通道使用率 > 0
v3.11  AI 生成质量冲刺（2 周，分两轮）
    │  第一轮：单文件/小多文件成功率 -> 80%
    │  第二轮：多文件 placeholder 推广 + token 策略
    ▼ 判定：标准源码成功率 >= 80%
v3.12  创作者留存闭环（1-2 周）
    │  仪表盘 + 点赞 + 站内通知
    ▼ 判定：创作者 7 天回访率可测量
v4.0 判定节点  看数据决定
    │  数据好 → 教程系列 / 搜索增强 / 邮件通知
    │  数据不好 → 继续打磨核心体验
```

---

## 3. 为什么这样排序

### 3.1 基础设施修复最先做

- Migration 不完整意味着每次加新表/新列都有部署翻车风险（06 号 Agent：最高阻塞等级）
- 埋点未调用意味着后续所有版本都无法用数据验证效果（04 号 Agent：先有数据再谈增长）
- 关键路径测试不足意味着改 AI prompt 无法快速知道是否回归（02 号 Agent：P1 技术债务）
- 这三项都是后续版本的前置条件，不做不行

### 3.2 首次体验先于 AI 质量冲刺

表面上看应该先修 AI 质量再引流新用户。但有两个理由让首次体验排前面：

1. **首次体验快速通道不依赖 AI 质量改进。** 它用的是 Redux 模板 + 现有生成能力。用户用模板生成时，Redux 代码是系统最熟悉的场景，成功率本身较高
2. **尽早上线才能收集数据。** `template_used`、`onboarding_started/completed` 这些事件需要真实用户触发。越早上线，v3.12 留存分析时的数据积累越多

但首次体验的版本主题不是"大规模获客"，而是"让愿意来的用户能更快看到成品"。AI 质量冲刺紧随其后，确保"来了之后不会失望"。

### 3.3 AI 质量冲刺分两轮

06 号 Agent 的代码分析表明，把所有 AI 质量问题打包成 1-2 周的冲刺大概率会失败：

- 单文件的 prompt 优化和 source-preprocessor 接入是低风险、可快速验证的
- 多文件 placeholder 推广涉及 `multi-phase-generator.ts` 的系统性改造，工程量更大
- 分两轮可以做"先交付可验证成果，再处理复杂场景"

### 3.4 创作者留存最后做

- 仪表盘需要 events 表有足够数据积累，所以排在首次体验 + AI 质量之后
- 点赞和站内通知需要新增数据库表（likes、notifications），需要 migration 基础设施可靠
- 邮件通知推迟到 v4.0 以后——域名验证 + email 未验证 + 用户基础极小（06 号 Agent：风险等级中高）

### 3.5 不先做增长的原因

04 号 Agent 明确指出：**留存优先于获客。在没有留存机制的情况下投入获客，等于用漏水的桶打水。** 当前 Retention 是 AARRR 最薄弱环节。v3.10-v3.12 的排序逻辑是"先把桶修好，再打水"。

### v3.10 前置：基础设施修复

**核心目标：** 让后续迭代可安全部署、可数据验证。

**为什么现在做这个：**
- 问题 32（migration 漂移）和问题 4（生产库缺表导致构建失败）是硬阻塞
- 6 个关键事件已注册但从未调用，v3.10 的所有"验证"都需要这些事件有数据
- 测试覆盖不足导致 AI prompt 迭代无快速反馈环

**要做哪些关键能力：**

| 项目 | 具体工作 | 估计时间 |
|------|---------|---------|
| Migration 基线修复 | 以当前 schema.ts 为准生成新 baseline；补齐缺失 SQL 文件；部署脚本加入 migration 回放 | 1 天 |
| 埋点补全 | 补全 6 个已注册未调用事件（onboarding_started/completed、template_used、generation_started/completed/failed）；新增 5 个关键事件注册（tutorial_published、tutorial_previewed、share_dialog_opened、tutorial_feedback_submitted、explore_search_zero_results） | 1 天 |
| 关键路径测试 | 为 assembler patch 应用链路补多步测试；为 validateTutorialDraft 补边界场景测试；为 API 端点补请求/响应契约测试 | 1-2 天 |
| 生成错误恢复统一 | 为生成失败定义结构化 errorCode 消费路径（前端统一改为 code 驱动的恢复分支） | 1 天 |

**不做哪些能力：**
- 不做新产品功能（不新增页面、不新增表）
- 不做 AI prompt 优化（留给 v3.11）
- 不做首次体验改造（留给 v3.10）

**对技术架构的要求：**
- 不涉及架构变更，纯基础设施层面
- 需要验证 `drizzle-kit push` 在生产环境可用

**主要风险：**
- Drizzle migration 重建可能暴露更多 schema 不一致（06 号 Agent 提示）
- 关键路径测试需要理解完整业务流程，工作量的不确定性高

### v3.10：首次体验破冰 + 阅读量可见化

**核心目标：** 让新用户 60 秒内看到成品，让创作者看到有人读。

**为什么现在做这个：**
- 首次体验门槛过高是漏斗最大断裂点（03、04 号 Agent 共识）
- 创作者发布后零反馈是留存最大杀手（01、03、04 号 Agent 共识）
- 两项工作都不依赖 AI 质量改进，可以提前交付

**要做哪些关键能力：**

| 项目 | 具体工作 | 估计时间 |
|------|---------|---------|
| 首次体验快速通道 | 首页增加"快速体验"按钮 -> 跳过 /new 表单 -> 用 Redux 模板一键创建+生成 -> 生成完成后默认进入预览模式 | 3-5 天 |
| 生成等待期预期管理 | 等待开始前展示成品效果截图/动画作为锚定 | 0.5 天 |
| 创作者阅读量可见化 | listDraftSummaries() 增加阅读量子查询；草稿列表卡片显示阅读次数（>0 才显示）；/u/[username] 显示总阅读量 | 1 天 |
| CTA 强化 | 首页 CTA 文案中文化；底部 CreateCTA 扩展到阅读过程中；携带 ref 参数归因 | 0.5 天 |
| 生成失败体验改善 | 展示可操作建议而非原始英文错误文本；区分"输入问题"vs"系统问题" | 1 天 |

**不做哪些能力：**
- 不做多步引导 overlay
- 不改编辑工作区布局
- 不增加新页面
- 不做 AI 模型选择 UI 改造（隐藏到高级设置是低成本优化，可以搭车做但不单独立项）

**对技术架构的要求：**
- `/new` 创建后跳转路由设计（问题 38 的解决方案）
- 生成中预览改为请求正式 payload API（清理旧的 streaming-preview route）
- events 表可能需要 `(eventType, createdAt)` 复合索引

**主要风险：**
- 快速通道的路由重构工作量被低估（06 号 Agent：3-5 天，而非 1-3 天）
- 如果当前 AI 生成成功率真的很低（< 50%），快速通道反而会暴露问题
- 模板只有 Redux 一种，非 React 用户可能觉得无关

**成功判定：**
- `template_used` 事件有数据写入
- 快速通道用户的"看到成品"完成率 >= 50%
- 创作者阅读量数据有展示

### v3.11：AI 生成质量冲刺（两轮）

**核心目标：** 把 AI 生成成功率从"碰运气"变成"大概率成功"。

**为什么现在做这个：**
- AI 生成不可靠是产品的生命线问题（01、03、04、05、06 号 Agent 共识）
- v3.10 的快速通道已经上线，需要可靠的生成能力支撑
- 种子内容批量生产需要高成功率

**第一轮（1 周）：单文件/小多文件场景**

| 项目 | 具体工作 | 估计时间 |
|------|---------|---------|
| Prompt 工程优化 | outline-prompt.ts 和 step-fill-prompt.ts 双路改进；针对高频失败模式做 prompt 针对性强化 | 2-3 天 |
| source-preprocessor 接入 | 已建但未接入 prompt 流的源码压缩/清理模块，正式接入生成链路 | 1 天 |
| parseJsonFromText 统一 | 问题 33：三处重复实现合并为一处（lib/ai/parse-json-text.ts） | 0.5 天 |
| 渐进式快照 placeholder 完善 | 将 placeholder stub 机制从渐进式快照推广到普通多文件输入 | 1-2 天 |

**第二轮（1 周）：复杂场景系统性改造**

| 项目 | 具体工作 | 估计时间 |
|------|---------|---------|
| Token 预算策略调整 | token-budget.ts 的预算策略优化，应对大源码场景 | 1-2 天 |
| Source coverage 崩塌修复 | 针对问题 35/36/37，确保 step-fill 不再把所有 patch 堆到主文件 | 2-3 天 |
| 生成前源码预检 | 语言推导、行数检查、结构分析，不合格给明确提示 | 1 天 |
| 质量迭代闭环 | 基于 npm run review:generation 的 review rubric 做回归验证 | 持续 |

**不做哪些能力：**
- 不做大源码分步生成（需要重新设计生成状态机，风险 70%——推迟到 v4.0）
- 不换模型架构（DeepSeek 8192 token 上限是硬约束，用 token 预算策略绕行）
- 不做用户 API 密钥管理（需要独立前置的加密基础设施，不在本轮范围）
- 不重构三阶段架构（outline -> step-fill -> validate 的流程不变）

**对技术架构的要求：**
- AI 层（lib/ai/）内改动为主，不触碰渲染链路和服务层接口
- source-preprocessor 接入需要修改 multi-phase-generator.ts 的输入准备阶段
- placeholder 推广需要修改 generate-tutorial-draft.ts 的 baseCode 构建逻辑
- SSE 协议事件类型建议提取共享类型定义（lib/schemas/sse-events.ts）

**主要风险：**
- 多文件 placeholder 推广比预期复杂——问题 37 指出普通多文件代码库同样丢失 targetFiles
- Token 预算策略对生成质量的影响需要在多种源码上反复验证
- 没有生成成功率的历史基线，不知道改进了多少
- 672 行的 multi-phase-generator.ts 接近单文件可维护上限，改动有回归风险

**成功判定：**
- 标准源码（单文件/小多文件）生成成功率 >= 80%
- 大源码（> 300 行）生成成功率 >= 60%
- 多文件教程 patch 文件覆盖率 >= 60%
- `npm run review:generation` 对 5 份不同源码跑 review 的可接受率 >= 4/5

### v3.12：创作者留存闭环

**核心目标：** 给创作者正反馈，让他们发布后愿意回来。

**为什么现在做这个：**
- v3.10 的首次体验带来了新用户和新教程
- v3.11 的 AI 质量让教程可靠可用
- 创作者需要看到自己的内容被消费，否则不会有第二次

**要做哪些关键能力：**

| 项目 | 具体工作 | 估计时间 |
|------|---------|---------|
| 创作者仪表盘 | 新增 /dashboard 路由；events 表聚合查询（总阅读量、7 天趋势、最受欢迎教程）；events 表加 (eventType, createdAt) 索引 | 3-4 天 |
| 教程点赞 | 新增 likes 表（tutorialId + userId 联合主键，one-way 不可取消）；API + 探索页/教程页 UI 展示 | 2-3 天 |
| 站内通知 | 新增 notifications 表；AppShell 通知图标（客户端轮询方案）；阅读里程碑通知 + 点赞通知 | 3-4 天 |
| 教程底部闭环 | 轻量反馈（yes/no，不需要评论）；底部推荐同标签教程 | 1 天 |
| 草稿续写引导 | 未完成草稿高亮、快捷入口 | 0.5 天 |

**不做哪些能力：**
- 不做邮件通知（域名验证 + email 未验证 + 用户基础极小——推迟到 v4.0）
- 不做评论系统（内容审核成本高，用户量不够）
- 不做实时推送（WebSocket，架构成本过高）
- 不做用户关注/粉丝体系
- 不做创作者收益/付费

**对技术架构的要求：**
- 新增 2 张表（likes、notifications），符合现有 schema 扩展模式
- 需要修改 ExploreTutorial 类型（lib/types/api.ts）加入 likeCount
- AppShell 当前是 Server Component，通知计数需要混合渲染方案（Server Shell + Client Notification Bell）
- events 表需要复合索引优化

**主要风险：**
- 图表库选型增加依赖（当前 package.json 没有）
- AppShell 渲染模式调整可能影响多个页面
- 仪表盘的零数据状态需要产品决策（"数据为零时展示什么"）
- 站内通知的轮询频率对单实例部署的负载影响未知

**成功判定：**
- 创作者 7 天回访率可测量
- 通知回访率 >= 10%（48 小时内）
- 教程反馈率 >= 5%

---

### v4.0 判定节点

v3.12 完成后看两个数据：

1. **周活用户是否在增长？** 连续 2 周周活下降 → 继续打磨核心体验
2. **是否有 >= 5 个活跃创作者？**（30 天内有创建或编辑行为）没有 → 回溯获客和激活环节

两个都是 → 可以规划 v4.0（教程系列、搜索增强、邮件通知、用户 API 密钥等高级功能）。不是 → 继续打磨当前环节，不做高级功能。

## 5. 明确列出"不做什么"

### 5.1 近期版本（v3.10-v3.12）明确不做

| 方向 | 不做的原因 | 重新评估时机 |
|------|-----------|------------|
| 大源码分步生成 | 需要重新设计生成状态机（风险 70%），当前用 placeholder 推广 + token 预算绕行 | v4.0 候选 |
| 邮件通知（Resend 集成） | 域名验证 + email 未验证 + 用户基础极小 | v4.0 候选 |
| 用户 API 密钥管理 | 需要独立的加密基础设施前置，不应和 prompt 优化混在一起 | v3.12 之后作为独立任务评估 |
| 评论系统 | 内容审核成本高，用户量不够时是鬼城 | v4.0 以后，用户量 >= 500 |
| 推荐算法 | 行为数据极薄，冷启动无解 | 积累 >= 1000 用户行为数据后 |
| 教程系列/合集 | 需要 >= 30 篇教程和活跃创作者 | v4.0 候选 |
| 实时协作编辑 | 架构成本极高，当前数据模型不支持多写者 | v5.0 以后，且只做"异步协作" |
| 付费/商业化 | 商业模式未验证 | 有 >= 100 活跃创作者后 |
| 移动端 App/PWA | 核心创作场景在桌面端 | 持续做响应式改进，不集中做 App |
| A/B 测试框架 | 用户量不够支撑统计显著性 | 有足够流量后 |
| 搜索增强（语义/向量搜索） | 内容量 < 30 篇时搜索没有意义 | 种子内容 >= 30 篇后 |

### 5.2 当前可做但不立项的低成本搭车项

以下事项不需要立项，但在相关版本开发时可以顺手做：

| 事项 | 搭车版本 | 估计时间 |
|------|---------|---------|
| AI 模型选择折叠到高级设置 | v3.10（首次体验改造时） | 0.5 天 |
| 教学风格选择折叠到高级设置 | v3.10 | 0.5 天 |
| SEO 基础补全（sitemap + canonical URL） | v3.10 | 0.5 天 |
| SSE 协议类型共享定义 | v3.11（AI 层改动时） | 1 天 |
| multi-phase-generator.ts 拆分 | v3.11（placeholder 推广时） | 2-3 天 |

## 6. 整体时序与节奏

```
v3.10 前置（基础设施修复）     3-5 天
    │
    ▼ 判定：migration 可靠 + 埋点有数据
v3.10（首次体验 + 阅读量）     1 周
    │
    ▼ 判定：template_used 事件有数据
v3.11（AI 质量冲刺 - 第一轮）  1 周
    │
    ▼ 判定：标准源码成功率 >= 70%
v3.11（AI 质量冲刺 - 第二轮）  1 周
    │
    ▼ 判定：标准源码成功率 >= 80%
v3.12（创作者留存）           1-2 周
    │
    ▼ 判定：创作者 7 天回访率可测量
种子内容 + 埋点积累期          1-2 周（与 v3.12 并行）
    │
    ▼ 看数据决定
v4.0 判定节点
```

**总估计：7-12 周**

**节奏控制原则：**
1. 每个版本结束时必须判定：达标还是不达标。不达标不进入下一版本
2. v3.11（AI 质量冲刺）是最关键的版本——如果生成质量上不去，后续都是空转。这个版本最不惜时间
3. 每个版本最多 2 周。超过 2 周说明范围太大或遇到意外复杂度，砍范围不延长时间

---

## 7. 增长停机条件

以下情况出现时，暂停新功能开发，回到核心体验打磨：

1. **首次体验成功率 < 50%。** 说明 AI 生成可靠性不足以支撑获客，应继续优化 prompt
2. **创作者 7 天回访率持续为零。** 如果阅读量可见化上线后创作者仍然不回来，问题在核心产品价值本身
3. **4 周内活跃创作者 < 3 人。** 产品还没有找到愿意反复使用的用户群体
