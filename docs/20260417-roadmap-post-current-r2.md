# VibeDocs 产品路线图 R2 (v3.10 → v4.0)

> 基于 6 份子 agent 分析的整合路线图。经 decision-log.md 共识提炼和冲突裁决后产出。日期 2026-04-17。

---

## 1. 系统现状定位

**VibeDocs 当前处于"核心链路跑通但用户价值未验证"阶段。** 产品有一个清晰的差异化定位（AI 驱动 scrollytelling 教程生成）和一套成熟的渲染引擎，但 AI 生成可靠性不足（估计 50-60%）、首次体验门槛过高（5 分钟 + 大概率失败）、创作者零留存机制、内容生态几乎为空。所有环节的转化率都不可测量——14 种埋点事件已定义但仅有 2 种被实际调用。

---

## 2. 路线图主线

**从"能跑"到"有人用"到"有人留"。** 每个版本解决用户生命周期中的一个断裂点：v3.10 让产品可观测并让新用户 60 秒内看到成品，v3.11 给创作者第一个回来的理由，v4.0 通过互动机制打破冷启动循环。每一环都是前一环的必要条件。

---

## 3. 分阶段路线图

---

### 第一阶段：v3.10 — 可观测 + 首次体验破冰

#### 核心目标

让产品从"盲飞"变成"有仪表盘飞行"，同时把新用户首次体验从 5 分钟压缩到 60 秒。

#### 为什么现在做这个

当前最大的矛盾是：产品有一个完整的创作链路，但没有人知道它是否好用，也没有人能快速体验到它的价值。14 种埋点事件已定义但未调用，所有产品决策都是猜测。新用户从到达首页到看到第一个成品需要注册 + 填 7 个字段 + 等待 30-60 秒 AI 生成（成功率约 50-60%），大部分人在到达"看到成品"之前就离开了。

#### 要做哪些关键能力

**工作流 0：基础设施修复（第 1 天，所有后续工作的前置）**

| 工作项 | 具体内容 | 工作量 |
|--------|---------|--------|
| Migration 基线修复 | 对齐 `_journal.json`、实际 SQL 文件和数据库状态，确认 0003 migration 已应用 | 1 天 |

**工作流 A：点亮埋点（1-2 天）**

| 事件 | 插入位置 |
|------|---------|
| `generation_started` / `generation_completed` / `generation_failed` | `POST /api/drafts/[id]/generate` 各阶段 |
| `tutorial_viewed` | `app/[slug]/page.jsx` 教程页面渲染 |
| `cta_clicked` | 首页"开始创建"按钮、ShareDialog 分享按钮 |
| `onboarding_started` | `app/new/page.tsx` 表单首次渲染 |
| `onboarding_completed` | `POST /api/drafts` 成功响应后 |
| `tutorial_created` | `POST /api/drafts/[id]/publish` 成功后 |

验收标准：上线 1 周内 events 表对上述全部事件有持续写入；能回答"每日有多少人使用产品"和"生成成功率是多少"。

**工作流 B：60 秒快速体验通道（4-6 天）**

- 首页增加"60 秒体验"按钮，点击后自动创建草稿 + 填充 Redux 模板数据 + 触发生成
- 生成完成后直接跳转预览模式（而非编辑模式）
- 预览页展示"这是 AI 自动生成的教程，你也可以创建自己的"引导文案
- 模型选择和教学风格选择在此流程中完全隐藏
- 处理匿名用户场景（如果快速体验不要求登录）

验收标准：`template_used` 事件触发；从点击按钮到看到教程成品 < 90 秒。

注意：实际工作量比表面估计（2-3 天）更高，因为需要处理跳过表单的创建流程 bypass 和匿名用户关联。

**工作流 C：生成错误体验改善（2-3 天）**

- 后端已定义 11 个结构化 `errorCode`，前端统一消费 `errorCode` 而非 `message.includes(...)`
- 消除 `use-generation-progress.ts`、`use-draft-workspace-controller.ts`、`classify-error.ts` 三处重复逻辑
- 区分"输入问题"（源码太短/太长/格式不对）和"系统问题"（AI 暂时不可用），给出不同引导
- 每种错误给出明确的"下一步操作"按钮

验收标准：前端不再有任何 `message.includes(...)` 的错误分支判断；生成失败时用户看到可操作的中文提示。

**工作流 D：种子教程补充（1-2 天，可与 A-C 并行）**

- 用 Redux 模板 + 几个经典库（Zustand、Vue Router 等）生成 3-5 篇高质量教程
- 通过 registry 机制注册，直接在首页和探索页展示
- 如果无法自动生成高质量教程（评分 < 90），手工编写 2-3 篇 DSL 数据

验收标准：首页和探索页不再出现"暂无已发布的教程"；至少有 5 篇可浏览的教程。

**工作流 E：阅读量可见化简化版（1-2 天，随埋点同步）**

- 草稿列表卡片显示阅读次数（> 0 才显示，避免"0 阅读"的心理打击）
- 阅读次数来源：events 表中 `tutorial_viewed` 事件的聚合
- 教程阅读页底部增加"你是作者？查看数据"的入口

验收标准：创作者打开草稿列表时能看到每篇已发布教程的阅读次数。

**持续并行：AI 生成质量微调**

- source-preprocessor.ts 接入生成链路（代码已存在但未接入，2-3 天）
- 轻量 prompt 调优，聚焦单文件/小多文件场景
- 每轮实验走 `npm run review:generation` 标准流程
- 不承诺版本级别的"完成"，用 generation-quality-loop 的量化停机条件衡量

#### 不做哪些能力

| 不做 | 原因 |
|------|------|
| AI prompt 大幅重构 | 需要 v3.10 的数据基线支撑决策，否则是盲目优化 |
| 完整创作者仪表盘 | 依赖埋点数据运行一段时间，否则仪表盘里没东西 |
| 教程系列/合集 | 高复杂度全栈功能，当前连基础留存都没有 |
| 评论系统 | 内容审核是持续运营成本，教程总量不足 |
| 编辑器内嵌预览 | 编辑体验优化属于"好用"层面，当前优先解决"能不能用" |
| 点赞/收藏 | 需要新增表和全栈功能，阅读量可见化是更低成本的起点 |
| 站内通知 | 当前没有足够的触发事件值得通知 |

#### 对技术架构的要求

- 不动分层结构，所有工作在现有层内叠加
- 埋点统一走 `lib/monitoring/analytics.ts` 的 helper，保持 fire-and-forget 模式
- 快速体验通道的新页面/API 先建 service，不要在路由里写业务逻辑
- errorCode 统一消费后需抽取为共享类型定义（`lib/types/` 或 `lib/schemas/`）
- Migration 基线修复后所有后续 schema 变更才能安全进行

#### 主要风险和缓解措施

| 风险 | 缓解措施 |
|------|---------|
| 快速体验通道工作量超预期（匿名用户处理） | 按 4-6 天排期，不按 2-3 天；如果匿名流程太复杂，第一版只支持登录用户 |
| source-preprocessor 接入效果不确定 | 先做小规模实验，用 generation-quality-loop 量化效果再决定是否推广 |
| 种子教程自动生成质量不够 | 允许手工编写 DSL 数据兜底 |
| Migration 基线修复发现更多不一致 | 预留 0.5 天缓冲 |

#### 预估工作量

不含持续并行的 AI 微调：10-14 天（约 2 周）。

#### 成功判定标准

| 指标 | 目标 | 当前基线 |
|------|------|---------|
| 生成成功率 | >= 70% | 未知（估计 50-60%） |
| 快速体验通道完成率 | >= 80% | N/A |
| 新用户"到达首页到看到成品"时间 | < 90 秒 | > 5 分钟 |
| 教程阅读 PV/日 | 可测量（有基线） | 不可测量 |
| 种子教程数量 | >= 5 篇 | 极少 |

### 第二阶段：v3.11 — 创作者留存闭环 + AI 质量迭代

#### 核心目标

给创作者"发布后回来"的第一个理由，建立最轻量的正反馈循环。同时基于 v3.10 积累的数据做 AI 生成质量的定向迭代。

#### 为什么现在做这个

v3.10 解决了"看不到"和"试不到"的问题，但创作者的体验仍然是"发布即失联"——系统不提供任何回访理由。没有阅读数据、没有互动通知、没有"你的教程有新读者"的触发。即使 v3.10 成功吸引了新用户并让他们完成了首次创作，这些创作者也不会回来。没有创作者留存就没有持续内容产出，没有内容就没有产品。

#### 前置条件（决策门）

- v3.10 的埋点数据已运行至少 2 周，有基线数据
- 生成成功率 >= 60%（否则优先做 AI 质量冲刺，推迟创作者留存工作流）
- 至少有 5 篇已发布教程（种子内容 + 早期用户产出）

#### 要做哪些关键能力

**工作流 A：创作者仪表盘（3-5 天）**

- `/dashboard` 页面：总阅读量、7 天趋势（折线）、最受欢迎教程 Top 5
- 数据来源：events 表聚合查询
- 仅创作者可见（有至少 1 篇已发布教程的用户）
- 前置：需要补充数据库索引——events 表 `(event_type, slug, created_at)` 复合索引

验收标准：创作者打开 `/dashboard` 能看到自己的教程被阅读的总次数和 7 天趋势。

注意：仅在有数据后上线。如果 v3.10 上线后 2 周内阅读量数据为零或极低，仪表盘继续推迟。

**工作流 B：教程阅读页底部增强（2-3 天）**

- 底部增加"相关教程"推荐区块（基于标签 SQL join，不引入推荐算法）
- 底部增加"你觉得这篇教程有用吗？"轻量反馈（yes/no，记录到 events 表）
- 底部 CTA 文案优化："想用你擅长的代码创建一篇教程？"

验收标准：底部推荐区块展示 >= 2 篇相关教程（在有足够内容时）；yes/no 反馈有实际点击。

**工作流 C：AI 生成质量迭代（持续，与 A-B 并行）**

- 根据 v3.10 积累的 `generation_failed` 事件分析，识别 Top 3 失败原因
- 针对 Top 3 原因做 prompt 调整或 source-preprocessor 效果验证
- 每轮实验走 `npm run review:generation` 标准流程
- 目标：生成成功率从 60-70% 提升到 80%+
- 聚焦单文件/小多文件场景的成功率，复杂场景暂不处理

验收标准：生成成功率比 v3.10 基线提升 >= 10 个百分点。

**工作流 D：数据库索引补充（0.5 天）**

- `events` 表：`(event_type, created_at)` 复合索引
- `published_tutorials`：`published_at` 降序索引
- `drafts`：`(userId, updated_at)` 复合索引

验收标准：仪表盘聚合查询响应时间 < 500ms。

**条件工作流 E：multi-phase-generator 拆分（3-5 天，仅在确认需要分片生成时做）**

- 如果 v3.10 的 source-preprocessor + prompt 调优效果不够，需要做大源码分步生成
- 按阶段提取 executor（outline-executor + step-fill-executor），用 lifecycle hooks 替代内联 SSE 推送
- 拆分后需补充集成测试验证行为一致性
- 拆分过程中不能破坏 SSE 协议兼容性

不做条件：如果 v3.10 的轻量优化已将生成成功率提升到 80%+，此工作流延后。

#### 不做哪些能力

| 不做 | 原因 |
|------|------|
| 点赞/收藏 | 需要新增表和全栈功能，仪表盘是更低成本的留存起点 |
| 站内通知系统 | 通知频率低，当前没有足够的触发事件值得通知 |
| 教程系列/课程包 | 高复杂度，依赖创作者留存数据验证 |
| 评论系统 | 内容审核是持续运营成本，教程总量不足 |
| 大源码分步生成 | 先观察 source-preprocessor 效果，效果不够时再做拆分 |
| 搜索引擎升级（pg_trgm 等） | 搜索不是留存问题，标签筛选已够用 |

#### 对技术架构的要求

- 仪表盘聚合查询需要 events 表索引支撑
- 仪表盘数据查询走新的 service + repository，不直接在页面中写 SQL
- 底部推荐用标签 SQL join，不引入推荐引擎
- multi-phase-generator 拆分保持 SSE 协议向后兼容
- SSE 协议类型在 generator 和前端之间共享，防止漂移

#### 主要风险和缓解措施

| 风险 | 缓解措施 |
|------|---------|
| 仪表盘上线后数据为零 | 提前检查 events 表写入量，数据不足则推迟仪表盘上线 |
| AI prompt 调优效果不确定 | 不承诺版本级别的完成，每轮实验量化评估 |
| multi-phase-generator 拆分引入回归 | 拆分前后用集成测试验证行为一致性，不破坏 SSE 协议 |
| 创作者回访率仍然极低 | v3.11 上线 3 周后评估，如果回访率 < 10% 则 v4.0 优先做更强的留存机制 |

#### 预估工作量

不含条件工作流 E：8-12 天（约 2 周）。
含条件工作流 E：11-17 天（约 2.5-3 周）。

#### 成功判定标准

| 指标 | 目标 | 衡量方式 |
|------|------|---------|
| 创作者 7 天回访率 | >= 20% | 发布后 7 天内再次访问 `/drafts` 或 `/dashboard` |
| 教程底部反馈率 | >= 5% | yes/no 点击 / 教程阅读 PV |
| 底部推荐点击率 | >= 3% | 推荐教程点击 / 教程阅读完成数 |
| 生成成功率 | >= 80% | `generation_completed` / `generation_started` |

### 第三阶段：v4.0 — 互动机制 + 内容增长

#### 核心目标

从"工具"升级为"平台"——通过读者互动机制给创作者更多正反馈，通过内容组织让已有内容更容易被发现，建立读者到创作者的转化循环。

#### 为什么现在做这个

v3.10 和 v3.11 解决了"能体验"和"能留存"的问题，但产品仍面临结构性困境：没有足够的内容，探索页对读者没有价值；没有足够的读者，创作者得不到足够的反馈。v4.0 通过互动机制（点赞）和内容组织（标签页增强、语言筛选）来打破这个循环。

#### 前置条件（决策门）

- v3.11 的创作者 7 天回访率 >= 15%
- 生成成功率 >= 75%
- 已发布教程总量 >= 15 篇（种子 + 用户产出）
- 埋点数据已运行 >= 6 周，有可靠的趋势数据

#### 要做哪些关键能力

**工作流 A：读者互动机制——点赞（5-7 天）**

- 新增 `likes` 表（tutorialId + userId 联合主键）
- 教程阅读页增加点赞按钮（需要登录）
- 点赞数在探索页作为排序维度
- 点赞时给创作者增加一个站内"新点赞"标记（最轻量的通知）
- 防刷机制：rate limiting（当前全局无 rate limiting）

验收标准：有互动（点赞）的读者的 7 天回访率 > 无互动读者；探索页"最多点赞"排序有实际使用。

注意：实际工作量可能达 7-10 天（含防刷和计数优化）。

**工作流 B：内容组织增强（3-5 天）**

- 教程标签的手动编辑入口从"仅发布后"扩展到"发布前后均可"
- 探索页增加按编程语言筛选（从 JSONB meta 中提取，SQL 层过滤）
- 标签页面增强：每个标签页展示该标签下的教程列表
- 教程阅读页增加"关注作者"入口（链接到作者档案页）

验收标准：探索页的按语言筛选有实际使用；标签页有实际流量。

**工作流 C：传播增强（2-3 天）**

- embed 代码片段加入"由 VibeDocs 生成"水印
- 分享链接加入 UTM 参数追踪
- 社交分享后的效果可追踪（通过 `cta_clicked` 事件的 `share` 类型）
- 首页增加创作者数量和教程数量的社会证明展示

验收标准：分享链接带来的新用户可追踪；embed 页面有产品标识。

**工作流 D：编辑体验从"能用"到"好用"（可选，3-5 天）**

这部分是锦上添花，不影响核心价值链，可根据开发资源决定是否纳入：

- 编辑器内轻量预览面板（右侧折叠面板，修改后立即看到代码变化）
- 保存自动快照（编辑操作自动创建快照，用户无需手动管理）
- AI 模型选择和教学风格选择折叠到高级设置

验收标准：编辑器内预览面板可展示当前步骤的代码变化效果。

#### 不做哪些能力

| 不做 | 原因 |
|------|------|
| 评论系统 | 内容审核是高复杂度引入，且需要持续运营投入；轻量反馈 + 点赞已能提供基础互动 |
| 教程系列/两级路由 | 路由层重构风险高，且当前没有足够的教程密度支撑系列化 |
| 推荐算法 | 教程量 < 100 时标签匹配已够，算法投入浪费 |
| 付费功能/订阅制 | 活跃用户 < 100 之前不考虑商业化 |
| 大源码分步生成 | 需要先验证 v3.11 的 source-preprocessor + prompt 调优效果 |
| i18n 国际化 | 用户群体验证后再扩展语言 |
| 移动 App | 核心体验不适合移动端 |

#### 对技术架构的要求

- likes 表新增需走 migration（依赖 v3.10 的基线修复）
- 全局 rate limiting 是点赞功能的前置（当前除 GitHub API 代理外无 rate limiting）
- 语言筛选需在 SQL 层实现 JSONB 过滤，不走应用层
- 标签编辑入口变更需同步更新 tag-editor 组件的使用条件
- embed 水印不影响 CORS header 和现有 embed 功能
- 编辑器预览面板如果做，需要评估是否需要新的 API 端点或可复用现有预览路由

#### 主要风险和缓解措施

| 风险 | 缓解措施 |
|------|---------|
| 点赞功能防刷不够 | 先做最小防刷（IP + userId 频率限制），后续迭代加强 |
| 教程总量仍不足 15 篇 | v4.0 开始前评估，不足则加大种子内容投入或降低创作门槛 |
| 两级路由需求提前出现 | 先用 UI 组件实现"上一篇/下一篇"导航，不改变 URL 结构 |
| 编辑器预览面板工作量大 | 标记为可选，资源不够则不做 |

#### 预估工作量

不含可选工作流 D：10-15 天（约 2-3 周）。
含可选工作流 D：13-20 天（约 3-4 周）。

#### 成功判定标准

| 指标 | 目标 | 衡量方式 |
|------|------|---------|
| 读者到创作者转化率 | >= 3% | 通过底部 CTA 注册并创建教程的读者 / 总读者 |
| 教程总量 | >= 30 篇 | 已发布教程总数 |
| 月活创作者 | >= 10 人 | 30 天内发布或编辑教程的独立用户数 |
| 分享链接转化率 | >= 5% | 通过分享链接到达的新用户 / 分享点击数 |
| 创作者 30 天回访率 | >= 30% | 发布后 30 天内再次访问的比例 |

## 4. 明确列出"不做什么"

以下方向在 v3.10 → v4.0 全程不做：

| 方向 | 排除原因 | 重新评估条件 |
|------|---------|-------------|
| 实时协作编辑 | WebSocket + OT/CRDT 复杂度与当前阶段不匹配 | 产品验证成功 + 团队规模增长 |
| 大规模 SEO/SEM 投入 | 核心价值未验证，引来的用户会流失 | v4.0 成功标准全部达成 |
| 商业化/付费功能 | 活跃用户 < 100，过早商业化阻碍增长 | 月活创作者 >= 50 |
| 移动 App | 核心体验（代码编辑）天然适合桌面端 | 产品定位扩展 |
| 推荐算法 | 教程量不足，标签匹配已够 | 教程 > 100 篇 + 用户行为数据管道就绪 |
| 评论系统 | 内容审核是持续运营成本，教程总量不足 | 教程 > 50 篇 + 审核方案确定 |
| 教程系列/合集 | 路由重构风险高 + 教程数量不足 | 教程 > 50 篇 + 创作者留存数据验证 |
| 邮件通知 | 用户基数太小，邮箱未验证率可能很高 | 用户基数 > 500 + 邮箱验证机制就绪 |
| i18n 国际化 | 中文用户优先，国际化不产生留存 | 中文市场验证成功 |
| 社区/论坛 | 内容和用户量都不够支撑社区运营 | 月活用户 > 1000 |

## 5. 版本间的决策门

### v3.10 → v3.11 的决策门

| 条件 | 结果 |
|------|------|
| 埋点运行 2 周 + 生成成功率 >= 60% + 种子教程 >= 5 篇 | 进入 v3.11（创作者留存） |
| 生成成功率 < 60% | v3.11 优先级调整为"AI 质量冲刺"，创作者留存工作流推迟 |
| 生成成功率 < 40% | 回退——暂停新功能开发，集中解决 AI 生成可靠性 |

### v3.11 → v4.0 的决策门

| 条件 | 结果 |
|------|------|
| 创作者 7 天回访率 >= 15% + 教程 >= 15 篇 + 埋点 6 周 | 进入 v4.0（互动+增长） |
| 创作者 7 天回访率 < 10% | v4.0 优先做更强的留存机制（提前做点赞/通知），推迟传播增强 |
| 教程总量 < 10 篇 | 暂缓 v4.0，加大种子内容投入或降低创作门槛（如支持 Fork 教程） |

### v4.0 → v4.1+ 的决策门

| 条件 | 结果 |
|------|------|
| 教程 >= 30 篇 + 月活创作者 >= 10 + 读者到创作者转化率 >= 3% | 评估 v4.1 方向（教程系列/评论/商业化三选一） |
| 教程总量仍 < 20 篇 | 加大种子内容投入 + 降低创作门槛，不进入 v4.1 |

| 版本 | 核心工作量 | 含缓冲后总工作量 | 目标交付周期 |

## 8. 技术债务分配

技术债务不是独立版本，而是分布在各版本中消化。

| 债务项 | 分配到的版本 | 理由 |
|--------|------------|------|
| Migration 基线修复 | v3.10 工作流 0（第 1 天） | 阻塞后续所有 schema 变更 |
| errorCode 前端统一消费 | v3.10 工作流 C | 直接服务于"生成错误体验改善" |
| 关键路径测试补全 | v3.10 工作流 C 期间 | 支持 AI prompt 迭代的回归验证 |
| SSE 协议类型共享 | v3.10 或 v3.11 早期 | 防止 generator 和前端接口漂移 |
| DB 索引补充 | v3.11 工作流 D | 仪表盘聚合查询需要索引支撑 |
| source-preprocessor 接入 | v3.10 持续并行 | AI 质量迭代的一部分，成本低 |
| multi-phase-generator 拆分 | v3.11 条件工作流 E | 仅在确认需要分片生成时做 |
| workspace-controller 拆分 | v4.0 工作流 D（如果做） | 编辑体验优化时顺便重构 |

## 9. 版本膨胀的刹车机制

以下信号出现时立即冻结新需求进入当前版本：

- 版本开发时间超过计划的 1.5 倍
- 版本范围内的工作流超过 6 个
- 任何一个工作流的新增文件超过 15 个
- 需要新增数据库表超过 2 张（v4.0 除外）

## 10. 每个版本必须持续做的事

不论版本主题是什么，以下工作在每个版本中持续进行：

1. **埋点数据回顾**：上线后 2 周回顾 events 表数据，验证版本目标是否达成
2. **AI 生成质量监控**：持续追踪生成成功率、patch 匹配失败率、重试率
3. **文档同步**：每个版本交付时更新 AGENTS.md、technical-handbook、tutorial-data-format 三份文档
4. **问题记录**：实施中的结构性问题、竞态问题、框架约束坑记录到 v3-implementation-issues.md
